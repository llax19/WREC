## prefetch-enabled WREC-H 说明

### 本次操作

- 解释 `prefetch-enabled WREC-H` 与当前主结果中 `wrec_prefetch_queue_depth=0` 的区别。

### 操作原理

- WREC-H 的 admission/eviction 版本只在 miss 发生后决定是否把当前 expert 放入 cache，或是否 bypass。
- prefetch-enabled WREC-H 额外允许策略在处理完当前 router event 后，根据 train trace 统计和在线 recent-history 分数，主动把当前未访问但预测高价值的 expert 提前加载进 cache。

### 结论

- prefetch-enabled WREC-H 是“带主动预取动作”的 WREC-H。
- 当前 trace 上该版本 prefetch waste 很高，收益不如只做 admission/eviction 的 WREC-H v1，因此主结果采用 `wrec_prefetch_queue_depth=0`。

## WREC-H optimization literature scan

### 本次操作

- 检索 MoE expert offloading、expert cache、proactive prefetch、cross-layer prefetch 和经典 cache admission/replacement 资料。
- 重点参考 MoE-Infinity、ProMoE、Fate、Fiddler、TinyLFU、ARC 的设计动机。
- 将资料启发映射到当前 WREC-H v1 的失败点。

### 操作原理

- MoE-Infinity 强调 sequence-level activation trace，而不是全局聚合 hotness。
- Fate 强调 cross-layer gate prediction、shallow-favoring cache、prefetch overlap budget 和 ARC-style recency/frequency balance。
- ProMoE 强调 proactive caching 要依赖中间结果预测，否则 reactive miss 仍在关键路径。
- TinyLFU/ARC 提供 admission 与 recency/frequency 自适应思路，可避免 cache pollution。

### 结论

- 当前 WREC-H v1 主要缺陷是统计过粗、per-layer budget 固定、prefetch 没有置信度和 overlap 约束。
- 优化优先级应为：
  1. 改成 sequence/request-level pattern matching，替代全局 train hotness。
  2. 加 per-layer cache allocation，浅层偏固定缓存、深层偏预测/预取。
  3. 用 TinyLFU/ARC 风格 admission 替代线性 score 的硬阈值。
  4. prefetch 只在 `benefit > transfer + waste risk` 且能被 overlap 时触发。
  5. 再考虑 WREC-C constrained planner，不应直接训练 predictor。

## WREC-H v1 粗粒度问题解释

### 本次操作

- 解释“WREC-H v1 太粗，本质是 train global hotness + online recent-history admission/eviction”的含义。

### 操作原理

- `train global hotness` 表示从整个 train trace 聚合每层 expert 访问频率，用全局热门程度作为 cache 决策先验。
- `online recent-history` 表示 eval replay 时只看最近已经发生的 expert 访问，用短历史调整 admit/evict。
- 这两类信号没有显式区分 request 类型、layer 差异、预测置信度和 transfer overlap。

### 结论

- 这句话的核心含义是：WREC-H v1 已经比纯 LRU 多了 workload 统计，但仍只用了较粗的全局热门和短历史信号。
- 后续优化应引入 sequence-level、layer-aware、confidence-aware 信号，而不是只调线性权重。

## cache budget 合理性复核

### 本次操作

- 复核固定使用 `1/2/4` experts/layer 作为 cache budget 的实验含义。

### 操作原理

- `1/2/4` 是强受限 cache setting，适合放大 locality、admission 和 eviction 策略差异。
- 真实部署中的 cache budget 通常来自总显存余量、每层 expert weight 大小、KV cache 预留和并发 batch，而不一定每层固定相同数量。
- 若只报告 `1/2/4`，容易把结论限制在 micro stress test，不能充分说明方法在真实 memory budget 下的优缺点。

### 结论

- `1/2/4` 应保留为压力测试和 oracle-gap 诊断，但不应作为唯一主表。
- 后续应补充 total-budget sweep 和 per-layer adaptive allocation，例如 `25%/50%/75%` resident experts 或按显存预算换算的 cache slots。

### 本次操作

- `prefill_decode` 输出 `step=0` 的 prefill router events，并输出 `step=1..N` 的 decode router events。
- 将单请求 router events 改为请求成功后再一次性写入，避免 `continue-on-error` 下出现半条请求污染 trace。
- 先运行 `n=1, max_new_tokens=2` smoke，再运行 `n=16, max_new_tokens=16` debug pilot。
- 删除定位 bug 时产生的失败 smoke 半成品，只保留成功 smoke 和正式 pilot 文件。
- 输出：
  - `logs/processed/wrec/decode_pilot/mixtral8x7b_dolly_decode_pilot_n16_new16_mem48_20260502.jsonl`
  - `results/wrec/decode_pilot/router_trace_stats_mixtral8x7b_dolly_decode_pilot_n16_new16_mem48_20260502.json`

  - `results/wrec/decode_pilot/mixtral8x7b_dolly_decode_pilot_n16_new16_mem48_summary_20260502.md`
  - `results/wrec/decode_pilot/mixtral8x7b_dolly_decode_pilot_n16_new16_mem48_summary_20260502.json`

### 操作原理

- prefill 用 prompt forward 采集全部 prompt token 的 router logits，并保留 KV cache。
- decode 采用 greedy token，每步只 forward 新 token，采集该 generated token 在 32 个 MoE layers 上的 router events。
- 逐 token decode 时不传完整 attention mask；否则 Transformers 的 Mixtral load-balancing 统计会把单 token router logits 与完整序列 mask 相乘，触发维度不一致。
- 请求内先用临时 buffer 累积事件和 layer/expert 计数，请求成功后再写入文件并合并统计；失败请求只进入 failure file。

### 结论

- decode trace pilot 成功：`16/16` requests，`0` failures。
- pilot 共 `742` prompt tokens、`256` decode tokens、`31936` router events。
- decode 部分为 `8192` router events，即每个 decode token `32` router events、`64` expert refs。
- prefill 与 decode 的 per-layer expert hotness 有差异：TV shift p50 为 `0.0795`，p90 为 `0.1275`，max 为 `0.1493`；top-2 overlap mean 为 `0.4688`。
- decode 不应直接假设与 prefill 分布一致，下一步应在 decode pilot trace 上单独跑 cache replay 或扩大到 `eval n=64`。

## decode token selection 说明

### 本次操作

- 澄清 decode pilot 中 greedy token selection 的含义，以及它和 vLLM decode sampling 的关系。

### 操作原理

- 当前 pilot 每步用 HuggingFace 模型输出的最后位置 logits 做 `argmax`，选出下一个 token。
- vLLM decode 阶段同样基于当前 token 和 KV cache 得到 logits，但最终 token 可由 greedy、temperature、top-p、top-k、beam 等 sampling 参数决定。
- WREC trace 关心的是给定 token 序列下的 router/expert access；如果 token selection 策略变了，后续 token 序列和 router trace 也可能变化。

### 结论

- 当前 greedy decode pilot 主要验证 decode trace 采集链路和确定性 workload，不代表所有 vLLM sampling workload。
- 它具备方法验证意义，但推广到 vLLM serving 应补充与目标 serving sampling 参数一致的 decode trace。

## decode-only cache replay

### 本次操作

- 从 eval decode trace 中筛选 `phase=decode`，生成 decode-only replay trace。
- 使用已有 train prefill trace 作为 `static_hot` 和 WREC-H/WREC-H2 reference。
- 在 decode-only eval trace 上运行 fixed total-budget cache replay。
- budget sweep 为 total resident expert fraction `12.5%/25%/37.5%/50%/75%`，对应 total slots `32/64/96/128/192`。
- 输出：
  - `logs/processed/wrec/decode_eval/mixtral8x7b_dolly_eval_decode_only_n64_new16_mem48_20260502.jsonl`
  - `results/wrec/decode_replay/wrec_total_budget_decode_only_mixtral8x7b_dolly_eval_n64_new16_20260502.json`
  - `results/wrec/decode_replay/wrec_total_budget_decode_only_mixtral8x7b_dolly_eval_n64_new16_20260502.csv`
  - `results/wrec/decode_replay/wrec_total_budget_decode_only_mixtral8x7b_dolly_eval_n64_new16_20260502.md`

### 操作原理

- decode-only trace 只保留 generated-token events，使 `stall_ms_per_input_token` 实际按 decode token 归一化。
- WREC-H/WREC-H2 仍使用 prefill train trace 构建先验，用来测试 prefill-prior 到 decode workload 的迁移。
- Belady 使用 decode eval future，作为 decode-only 上界。

### 结论

- decode-only workload 为 `64` requests、`1024` decode tokens、`32768` router events、`65536` expert refs。
- WREC-H 相对 uniform LRU 只有小幅正收益：total slots `32/64/96/128/192` 下分别为 `9.20%/1.27%/1.62%/2.47%/6.86%`。
- WREC-H2 在 slots `64/96/128/192` 下弱于 LRU，gain 分别为 `-5.68%/-8.48%/-8.97%/-7.33%`。
- static-hot from train prefill 在该 small decode workload 上很有竞争力，说明 prefill hotness prior 有部分迁移性。
- Belady 明显优于所有可部署策略，说明 decode 存在可利用 locality，但当前 WREC-H2 的 request/cross-layer 权重不适合直接迁移到 decode。
- 当前结果不支持直接进入正式 decode train/eval；应先做 decode-specific tuning 或比较 decode-train prior。
