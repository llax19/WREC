## Mixtral INT8 HRM

- 使用 `/root/WREC/scripts/moe_affinity` 中已有 HRM 脚本，没有新增 `scripts/wrec` 代码。
- 当前机器为 `2 x NVIDIA GeForce RTX 3090 Ti`，每卡约 `24564 MiB`。
- 按 INT8 `1 byte/param` 估算，Mixtral expert FFN 权重约 `42.00 GiB`，总权重约 `43.50 GiB`。
- 双 3090 Ti 聚合显存下，INT8 Mixtral 处在容量临界点。
- 若 KV/cache 预算只有 `4 GiB` 且 runtime 能均匀放置权重，全权重驻留可行，瓶颈转为 GPU FFN compute/memory。
- 若 KV/cache 预算达到 `8 GiB` 及以上，专家无法全驻留，少量 miss 就会让 PCIe CPU-GPU expert transfer 成为主瓶颈。
- 单卡 24 GiB 口径下，INT8 Mixtral 仍是 expert-transfer-bound。
- 简洁报告写到 `WREC/results/wrec/hrm_eval/mixtral/hrm_mixtral8x7b_int8_2x3090ti_bottleneck_20260510.md`。

## Qwen single-GPU HRM

- 搜索本地 Qwen 模型与既有 HRM 输出。
- 当前 `/root/workspace/qwen1.5-MoE-A2.7B/config.json` 不存在，因此无法干净复跑 HRM evaluator。
- 使用既有 HRM 脚本产物 `WREC/results/wrec/hrm_eval/qwen1.5/hrm_policy_eval_qwen1_5_moe_a2_7b_single_gpu_calibrated_20260428.json` 提取单卡瓶颈结论。
- Qwen1.5-MoE-A2.7B bf16 总权重约 `26.66 GiB`，其中专家 FFN 约 `24.75 GiB`，dense/non-expert 约 `3.46 GiB`。
- 单卡 24 GiB 下不能全权重常驻，需要在 GPU 驻留比例、KV 位置、CPU-GPU 传输之间取舍。
- HRM 搜索中 `756` 个 policy 有 `169` 个因 GPU 显存不可行；可行 policy 中多数瓶颈为 `cpu-gpu-transfer`。
- 最优吞吐 policy 只让约一半 FFN 权重驻留 GPU，并将 attention/KV 放 GPU，瓶颈转为 GPU compute/memory。
- 若选择低 GPU weight residency 或较大 batch，CPU-GPU transfer 会迅速成为主瓶颈。

## Mixtral BNB state-only serving

- 用户启动并跑通 `mixtral-bnb-wrec-stateonly` 后，检查 vLLM `/v1/models`、WREC sidecar `/metrics` 和 GPU 显存。
- vLLM served model 为 `mixtral-bnb-wrec-stateonly`。
- Sidecar 记录到 `736` router events、`1472` expert refs、`307` shadow hits、`1165` shadow misses、`948` would admit、`217` would bypass、`948` would evict，final resident 为 `64`。
- GPU snapshot 为 GPU0 `19853 MiB / 24564 MiB`，GPU1 `10 MiB / 24564 MiB`。
- 结论是 Mixtral bitsandbytes 单卡真实 vLLM serving + WREC sidecar state-only export 已跑通。
- 该结果完成了此前 TP2 FP16 CPU-offload 路线未能完成的真实 runtime event-export 验证。
- 该结论只限定为 state-only runtime integration，不能声称已经控制真实 expert loading，也不能声称 finite-slot 已跑通。

## Mixtral BNB expert-residency state-only

- 用户完成 `WREC_EXPERT_RESIDENCY=1` 但不设置 slot capacity 的实验后，检查 vLLM `/v1/models`、WREC sidecar `/metrics` 和 GPU 显存。
- vLLM served model 为 `mixtral-bnb-wrec-residency-stateonly`。
- Sidecar 记录到 `1472` router events、`2944` expert refs、`614` shadow hits、`2330` shadow misses、`1896` would admit、`434` would bypass、`1896` would evict，final resident 为 `64`。
- GPU snapshot 为 GPU0 `19853 MiB / 24564 MiB`，GPU1 `10 MiB / 24564 MiB`。
- 该实验在上一阶段 scheduler sidecar export 基础上，额外启用 FusedMoE expert-residency manager。
- 因未设置 `WREC_EXPERT_RESIDENCY_SLOT_CAPACITY` 和 `WREC_EXPERT_RESIDENCY_ROW_COPY`，该路径仍为 state-only，不替换 GPU expert tensor，也不复制完整 expert weights。
- 结论是 Mixtral bitsandbytes 单卡真实 vLLM serving + WREC expert-residency state-only 已跑通。
- 这进一步验证了 `WREC_EXPERT_RESIDENCY=1` 不破坏真实 serving 路径。

## Overall conclusion

- 5 月 10 日的主线是从 HRM 容量判断推进到 Mixtral bitsandbytes 真实 serving state-only 验证。
- Mixtral INT8/BNB 在双卡聚合或单卡量化 serving 下已经能支撑真实 routed expert event export。
- 当前最强结论是 runtime event export 与 expert-residency state-only manager 都已在真实 Mixtral BNB serving 路径中跑通。
- 当前仍不能声称 finite-slot 已成功，也不能声称 WREC 已经实际控制 vLLM 的专家权重加载。
- 目前在实验上的下一步打算使用单卡3090Ti去跑qwen模型，这样既能避开vllm的量化开销，还能缓解内存压力
