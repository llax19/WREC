### 操作原理

- WREC-C 在需求 miss 时把 `resident + incoming expert` 放进一个 constrained target-set selection，受 total slots、可选 per-layer min/max 约束。
- WREC-C 的主动预取版本在 event tail 触发 bounded candidate planning，候选来自当前 resident、train-hot experts、online recent/request experts 和 cross-layer transition experts。
- 初版全局 256 候选每次重算过慢，因此改为 bounded candidates，并新增 `wrec_c_replan_interval` 控制 planner overhead。

### 遇到的困难与处理

- 初版 WREC-C 每次 miss 对全部 `num_layers * num_experts = 256` 候选重算 score，正式 replay 长时间未完成。
- 处理方法：中止慢任务，将 planner 改为 bounded candidate set；主动预取再加 `replan_interval=16`，避免每个 router event 都重排。
- 第一次加入 prefetch 后仍偏慢，说明 WREC-C 的 runtime overhead 后续必须进入 Phase 9 评估。

### 结论

- WREC-C 实现已完成并通过 smoke。
- no-prefetch WREC-C 与 WREC-H2 完全一致：total slots `32/64/96/128/192` 下 gain 都为 `67.10%/73.49%/78.98%/80.92%/76.91%`，waste 为 `0`。
- constrained-prefetch WREC-C 在 `queue=1, overlap=10ms, interval=16` 下弱于 WREC-H2，并产生明显 waste；例如 slots `32` 下 gain 从 H2 的 `67.10%` 降到 `61.65%`，waste 为 `664024112.13 bytes/token`。
- 当前不应把 WREC-C 作为主方法替代 WREC-H2；主方法应暂定 WREC-H2，WREC-C 作为 constrained planner ablation / negative result。

## WREC-C 预测精度改进方向

### 操作原理

- WREC-C prefetch 的失败不是简单的命中率问题，而是低置信预取导致 cache pollution 和 waste。
- 预取策略应优化 expected net benefit，优先提高 precision，而不是追求覆盖更多未来 expert。

### 结论

- 优先做 next-use/horizon 诊断，找出哪些特征能可靠预测 `H` 个 event 内会再次访问。
- WREC-C prefetch 应加入 confidence gate、do-no-harm eviction guard 和 horizon-specific predictor。
- 若仍不足，再训练轻量 ranker；ranker 目标应是 `benefit > pollution + waste`，而不是单纯预测 expert 是否会出现。

## WREC policy overhead 测量

### 本次操作

- 新增 `workspace/scripts/wrec/benchmark_policy_overhead.py`。
- 使用 Mixtral Dolly 主实验 trace 运行 WREC-H2 no-recent 配置的在线 policy overhead benchmark。
- 输出结果：
  - `workspace/results/wrec/calibration/wrec_policy_overhead_mixtral8x7b_dolly_n256_slots64_20260503.json`
  - `workspace/results/wrec/calibration/wrec_policy_overhead_mixtral8x7b_dolly_n256_slots64_20260503.md`

### 操作原理

- benchmark 只把在线 loop 作为核心 overhead：
  - cache lookup / touch；
  - `update_wrec_history` 在线状态更新；
  - WREC admission / eviction 决策。
- trace 加载和离线 trace prior 构建单独计时，不混入在线 policy overhead。
- 使用与主结果一致的配置：
  - eval trace: Mixtral Dolly eval n256；
  - train trace: Mixtral Dolly train n512；
  - total slots: `64`；
  - history size: `8`；
  - recent weight: `0`；
  - request weight: `1024`；
  - cross-layer weight: `1024`；
  - bandwidth: `41.37 GB/s`。

### 结果

- 处理 expert refs: `905408`。
- 对应 router events: `452704`。
- Demand misses: `147884`。
- WREC 在线 loop 总耗时: `13.857839 s`。
- 平均 overhead:
  - `15.306 us/expert ref`；
  - `30.611 us/router event`；
  - history update: `0.565 us/expert ref`；
  - admission decision: `88.290 us/miss`。
- Mixtral expert bytes: `352321536`。
- 在 `41.37 GB/s` 下，单次 expert transfer 估计为 `8.516353 ms`。
- policy loop 与单次 expert transfer 的比例约为 `0.001797`。

### 结论

- 当前 Python 实现的 WREC 在线决策开销相对 expert transfer 代价小两个数量级以上。
- 这支持将 WREC replay 表述为有硬件量级校准的 trace-driven replay，而不是只靠抽象 miss count 的 toy simulator。
- 该结果仍不是端到端 serving latency 证明；它只说明 policy CPU overhead 不太可能成为当前 transfer-bound proxy 的主要反证。

### 记录

主贡献代码没有形成稳定系统模块
现在真正有主线价值的是 WREC scoring、admission/eviction、total-budget cache replay 这一部分。但它目前主要包在实验脚本里，例如 trace 构建、cache simulation、ablation、summary、figure generation 等。答辩委员会质疑：

WREC 是一个方法，还是一组脚本跑出来的结果？
是否存在清晰的 CacheManager / Policy / RuntimeState 接口？
换 workload、换模型、换 serving trace 时，是否仍可复用？
哪些代码是方法本身，哪些只是实验 glue code？
如果不能回答，论文会显得“工程主线薄，实验脚本厚”。

生产环境验证不足
当前最强结果是 prefill-stage 的 offline replay：WREC-H2 在 fixed total-budget 下显著优于 LRU，并且有 Belady oracle gap、消融和带宽敏感性支撑。但这仍不是生产验证。生产环境至少会追问：

真实 vLLM / MoE runtime 中，expert weight cache 是否真的能被 WREC 控制？
WREC 决策是否能在线生成，CPU overhead 是多少？
expert transfer 是否能异步化、是否能 overlap？
如果预测错误，会不会造成 cache pollution 或 tail latency 变差？
decode 阶段失败后，系统如何 phase-aware fallback？


我作为答辩老师会重点追问

你的 WREC 到底优化的是 TTFT、TPOT、真实 latency，还是 simulator 中的 stall proxy？
如果只是 stall proxy，论文必须明确写：本文评估的是硬件校准的 trace-driven cache replay，不是端到端 serving latency。

为什么 WREC-H2 在 prefill 有效，但 decode 失败？
这不是小问题。它说明你的方法具有 phase-specific 假设。你应主动把它写成边界：prefill trace 中 request-local 与 cross-layer locality 稳定，decode 阶段 hotness shift 明显，需要 decode-specific prior 或 phase-aware policy。

WREC 是否只是“调参后的 popularity caching”？
你需要用消融回答：train-window-only 接近 LRU，说明单纯 global prior 不够；cross-layer 和 request-local 信号才是主要贡献；recent term 反而不稳定。这个逻辑要写清楚。

为什么不直接用 Belady / route-window oracle？
要强调 Belady 是理论上界，route-window prefetch 含未来信息或产生大量 waste，不是 deployable online baseline。WREC 的价值在于 online、无 routing 改动、零 prefetch waste 的 cache decision。

最建议补的工作

优先级最高的是三件事，不建议继续堆同类表格。

把 WREC 抽成方法模块
至少形成类似结构：

wrec/
  policy.py          # WREC score / admission / eviction
  cache_manager.py   # resident set, budget, victim selection
  trace_replay.py    # 调用 policy 的离线 replay
  metrics.py         # miss/stall/waste/oracle gap
这样答辩时可以说：实验脚本只是驱动器，核心方法是可复用的 expert cache policy。

补 policy overhead 和 transfer calibration
你需要证明 simulator 不是纯玩具：

WREC 每个 router event 的 CPU 决策开销；
Mixtral expert-size 对应的 CPU-GPU copy 时间估计；
当前 41.37 GB/s 带宽参数来自实测；
带宽下降时 saved stall 增大，符合 transfer-bound 设定。
做最小 runtime shadow mode
不一定立刻完整接管 vLLM expert cache，但至少可以做：

真实 runtime 产生 routed expert events
-> WREC shadow policy 在线消费 events
-> 输出 would-admit / would-evict / cache-state metrics
-> 与离线 replay 对齐
这能显著降低“只在离线脚本里成立”的质疑。
