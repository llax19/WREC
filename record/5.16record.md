## online_loop 主耗时判断

- 时间：`2026-05-16`（记录写入 UTC: `2026-05-15T12:25:39Z`）
- 操作：结合复跑结果与当前 `runtime_engine.py` / `policy.py` 热路径，重新判断 `online_loop_us_per_router_event` 的主要来源。
- 原理：`online_loop_us_per_router_event` 统计的是 `_process_event()` 整段循环时间，不只是一处函数。当前代码里它覆盖了 `ExpertRef` 对象构造、cache hit/miss 分支、`_update_history()`、`policy.admit_or_bypass()`、decision 字典构造等全部 per-event 工作。特别是在 miss 路径里，`policy.choose_victim()` 仍会扫描整个 `resident` 集合并对每个候选重复调用 `score()`，这部分复杂度仍然随 slot 数增长。
- 结果：现有数据表明 `online_loop_us_per_router_event` 的确还是主耗时口径，但它不是单一热点；它主要是“每个 router event 下的整条 Python 主循环”叠加出来的。删掉 resident 集合复制后 only 带来约 `3.6%` 的 loop 改善，说明后续收益更可能来自减少 per-event Python 工作量，而不是继续修补单个常数项。
- 结论：下一步最有价值的方向有三类：1) 减少 `_process_event()` 的调用/对象构造粒度；2) 缩短 miss 路径里 `choose_victim()+score()` 的候选扫描；3) 如果调用方不依赖完整 decisions，减少 per-expert decision payload 构造。

## victim 候选集限流实现

- 时间：`2026-05-16`（记录写入 UTC: `2026-05-15T12:32:39Z`）
- 操作：实现“`choose_victim()` 不扫全量 resident，只看较小候选集”的近似。修改了 `scripts/wrec/policy.py`、`scripts/wrec/runtime/runtime_engine.py`、`scripts/wrec/runtime/runtime_sidecar.py`、`external/vllm-0.19.0/vllm/v1/core/sched/wrec_sidecar_client.py`、`scripts/runners/run_qwen_real_inference_main.sh`。
- 原理：新增 `WrecPolicyConfig.victim_candidate_limit`。当该值大于 `0` 时，`choose_victim()` 不再对全量 `cache.resident` 打分，而是优先从 `cache.lru_order` 中取“最老的前 N 个”作为候选，再在这 N 个上按原 WREC `score()` 选最差 victim。这样仍保留了“先看老项再做 score 判别”的语义，但把 miss 路径的候选扫描复杂度从 `O(total_slots)` 压到 `O(N)`。
- 配置：runtime/sidecar/inproc client/runner 都已接入该参数，当前默认值设为 `32`。因此后续从 runner 发起的 sidecar/inproc 实验会默认启用“只看最老 32 个 victim candidates”；若要回退到全量扫描，可把 `SIDECAR_VICTIM_CANDIDATE_LIMIT=0`。
- 困难：1) 这个近似会改变策略语义，不能直接写死在 policy 内部，所以改成显式配置项并把最终值写进 metrics。2) `lru_order` 只在 runtime cache 上存在，因此候选裁剪需要在“有 LRU 顺序”和“没有 LRU 顺序”两种情况下都能退化工作。
- 验证：`python -m py_compile` 检查修改过的 Python 文件通过；`bash -n scripts/runners/run_qwen_real_inference_main.sh` 通过。补了一个最小内存检查，确认 `victim_candidate_limit=2` 时 `_trim_victim_candidates()` 确实只返回 `lru_order` 里最老的前两个 resident。
- 结论：这一步已经落地，且默认会影响后续 runtime 复跑。下一步应直接对 long prompt case 做 A/B：`victim_candidate_limit=32` 对比 `0`，重点看 `decision_us_per_miss`、`online_loop_us_per_router_event` 和 `shadow_miss_rate` 是否出现可接受的 tradeoff。

## WREC 主线 story 梳理

- 时间：`2026-05-16`（记录写入 UTC: `2026-05-16T00:00:00Z`）
- 操作：围绕当前两条核心实验线重新梳理项目 story：一条是 WREC 打分公式，一条是 selective prefetch runtime 实现。
- 原理：在现阶段，实验计划不能再从 runner 参数或历史失败方案出发，而应从最终要证明的核心论点反推。WREC 的方法价值不在于“维护一个更复杂的 shadow cache”，而在于利用 routed experts 的可预测性，把一小部分高价值 expert row 从 CPU offload backing 提前搬到 GPU，从而减少真实 serving 中的同步 expert copy。
- 结果：
  - story 的第一个支点是“expert 访问并非不可预测”，而是同时具备静态热门性、request-local recurrence 和 cross-layer correlation；WREC 打分公式就是把这些信号统一成一个“预取/驻留是否值得”的价值函数。
  - story 的第二个支点是“有了分数还不够，必须把分数变成真实的 latency gain”；WREC 主线 selective prefetch 通过复用 vLLM CPU offload backing、在 attention 与后续 MoE 之间触发异步预取、并用共享 residency budget 承载高分 experts，把打分结果转化成实际的 transfer overlap 与 sync-copy 减少。
  - 因此最终 story 不是“WREC 学会了完美预测下一个 expert”，而是“WREC 证明了 routed experts 的部分可预测性已经足够被系统利用，在受限 GPU 预算下换来真实 serving 收益”。
- 结论：后续正式实验应直接服务三个 claim：1) WREC score 比简单静态/局部启发式更能识别高价值 experts；2) WREC 主线能把这些分数转成更高 hit、更低 sync copy；3) 这些机制收益最终体现在 prefill 与 decode 的真实 serving 指标上。

  - 当前实现会在每个 attention 前调用 `maybe_prefetch_for_attention()`，按 ranking 预取最多 `active_slots` 个专家；见 `external/vllm-0.19.0/vllm/model_executor/layers/fused_moe/wrec_expert_residency.py:198-230`

- 结论：
  - 本轮首要发现不是“WREC 主线无命中”，而是“当前 shared-slot residency 实现没有把命中有效转化成 decode 收益”。
  - decode 退化的主因更接近 residency 生命周期和跨层共享策略，而不是 WREC score 或 ranking-only sidecar 的计算开销。
  - 下一步若要修 decode，优先方向应是减少 decode 上的无效预取与跨层 owner 抖动，而不是继续优化 sidecar 打分。

## decode 退化的修复方向分析

- 时间：`2026-05-16`
- 操作：继续检查 `external/vllm-0.19.0/vllm/model_executor/layers/fused_moe/wrec_expert_residency.py` 中的 shared-slot pool 初始化、owner 切换与 `active_slots` 规则，并结合 `runtime_qwen_real_table_b_full_n64_20260515T172203Z/wrec_residency_stats.jsonl` 做 decode / prefill 分相统计。
- 原理：要判断该怎么修，必须先区分“纯软件开销”“slot 预算不够”“decode 预取粒度不对”“共享 pool 生命周期设计有问题”这几类可能性。当前 partial 数据已经表明 sidecar ranking-only 的 Python 开销很低，因此重点应放在 residency manager 的生命周期与预取策略。
- 结果：
  - `maybe_prefetch_for_attention()` 会在每个 attention 层都触发预取；`begin_forward()` 会在对应 MoE 层等待 prefetch 完成并补同步 miss。
  - `_shared_slot_pool_key()` 只按 `slot_capacity / weight shape / dtype / device` 分组，不含 layer id，因此同形状层共享同一 slot pool。
  - `_claim_shared_slot_pool()` 在 owner 切换时一定会 `_invalidate_layer_slots(previous_state)`，因此前一层的 resident set 和 `last_prefetched_experts` 会被直接清空。
  - 当前未设置 `WREC_EXPERT_RESIDENCY_ACTIVE_SLOT_ALPHA`，所以 `active_slots` 在 decode 也恒等于 `slot_capacity=32`，即每层 decode 仍尝试预取约 32 个候选 expert；而 decode 实际 `U=4`。
  - partial decode 统计：
    - `rows_num_tokens_1=13272`
    - `sum(H)/sum(U)=0.5264`
    - `avg_prefetched_len=31.1657`
    - `sum_waste_transfer_bytes=6,672,982,474,752`
    - 各层 decode `avg_M` 都在 `1.69~1.91` 之间，没有哪层已经收敛到接近零 miss
- 结论：
  - 最低成本且最值得先做的修复不是“重写 score”，而是先把 decode 预取从“32 个大候选集”改成更小的 decode 专用预算，或直接在 decode 关掉预取。
  - 即便把 decode 预取集缩小，跨层 owner 切换导致的 resident 生命周期过短仍然存在；这意味着“用一个共享 pool 同时覆盖所有层并希望 decode 跨 token 复用”在当前设计下并不现实。
  - 因此修复优先级应分两层：
    - 第一层：用最小改动止血，避免 decode 上的大量无效预取。
    - 第二层：如果仍希望 decode 获益，需要重构 shared-slot 生命周期，而不是继续调 ranking 阈值。

## active slots 与 decode token 数核对

- 时间：`2026-05-16`
- 操作：核对 `attention.py`、`fused_moe/layer.py`、`wrec_expert_residency.py` 中 `num_tokens -> active_slots` 的传递链，并统计 `runtime_qwen_real_table_b_full_n64_20260515T172203Z/wrec_residency_stats.jsonl` 里的 `(num_tokens, active_slots)` 组合。
- 原理：用户希望 prefill / decode 都按“一次 forward 实际处理的 token 数”动态决定 `active_slots`。这需要先确认代码是否已经支持该逻辑，以及当前实验为什么 decode 仍是 `active_slots=32`。
- 结果：
  - attention hook 传给 `maybe_prefetch_for_attention()` 的是 `query.shape[0]`；MoE forward 传给 `_active_slots_for_forward()` 的是 `router_logits.shape[0]`。因此两边拿到的都是“本次 forward 实际处理的 token 数”。
  - 但 `_active_slots_from_state()` 只有在 `active_slot_alpha > 0` 时才按 `ceil(alpha * top_k * num_tokens)` 动态计算；当前默认 `active_slot_alpha=0`，因此直接退化成 `active_slots = slot_capacity`。
  - 本轮 partial `Table B` 统计中，所有 decode 行都是 `(num_tokens=1, active_slots=32)`；说明代码虽然支持动态 active slots，但当前配置没有启用。
  - 由于本轮 runner 设的是 `MAX_NUM_SEQS=1`、`MAX_CONCURRENCY=1`，所以当前实验里的 decode 确实是“一次 forward 只处理一个 token”。这不是 decode 一般规律，而是本轮实验设置导致的。若并发请求数更高，decode forward 可以同时包含多个请求各自的一个 token，此时 `num_tokens` 会等于该 microbatch 中的活跃 decode token 总数。
- 结论：
  - 用户想要的“按 forward token 数动态重定 `active_slots`”机制，代码里已经有，只需启用 `WREC_EXPERT_RESIDENCY_ACTIVE_SLOT_ALPHA>0`。
  - 若 `top_k=4` 且设 `ACTIVE_SLOT_ALPHA=1`，则本轮实验下 decode 会自然变成 `active_slots=4`，而 prefill 仍会因为 `min(slot_capacity, alpha * top_k * num_tokens)` 被截到 `32`。

## 结构性限制进一步核对

- 时间：`2026-05-16`
- 操作：继续核对 `qwen2_moe.py` 中 decoder layer 的执行顺序，以及 attention hook 与 MoE residency hook 在这一顺序中的相对位置。
- 原理：如果要判断“当前结构是否根本不适合解决 decode expert offload”，关键不是只看 miss/hit 数，而是看预取窗口到底有多大，以及 slot residency 能否跨 token、跨层稳定存在。
- 结果：
  - `Qwen2MoeDecoderLayer.forward()` 的顺序是 `self_attn -> post_attention_layernorm -> mlp`，中间没有额外长窗口。
  - attention hook 在 `Attention.forward()` 内部执行 `maybe_prefetch_for_attention(self.layer_name, query.shape[0])`；MoE hook 在 `Qwen2MoeSparseMoeBlock.forward_native()` 里执行 `begin_forward(self, router_logits)`。
  - 因此当前 WREC 的可重叠窗口，本质上只有“同一 decoder layer 内，从 attention 结束前后到该层 MoE 开始前”的一小段计算。
  - 在 decode 单 token 条件下，这段窗口比 prefill 短得多，而共享 slot pool 又在跨层切换时清空前一层 resident set，所以当前结构几乎同时失去了两件事：1) 没有足够长的 layer-local overlap window 去隐藏 copy；2) 没有足够稳定的跨 token same-layer residency 去积累 decode hit。
- 结论：
  - 当前结构更像“prefill 友好、decode 不友好”的 selective prefetch 结构。
  - 如果论文需要写结构性边界，这里已经有足够强的证据支持一个保守判断：在单共享 slot pool、单层短窗口预取、跨层 owner 失效这三点同时成立时，decode expert offload 很难获得稳定收益，哪怕 ranking 信号本身是有信息量的。
