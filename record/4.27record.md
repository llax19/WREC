### 操作原理

本次判断按照“先确认瓶颈是否存在，再决定是否设计优化方法”的顺序进行：

1. 当前 Qwen1.5-MoE-A2.7B 参数量较小，且现有服务端实验显示 request sorting 无法稳定超过

3. 原方案同时包含 trace、gate signal、cache、prefetch、evict、offload、quality penalty 和

1. 前置瓶颈判定：用 HRM/roofline-style 分析加小规模 profiling，选择真正 expert weight
   movement 成为瓶颈的模型与硬件配置，例如 Mixtral 8x7B、Mixtral 8x22B、DBRX 或更大的
   DeepSeek-MoE 系列，而不是只依赖 Qwen1.5-MoE-A2.7B。
2. 核心方法闭环：只做“route-trace/workload-aware expert cache replacement + prefetch”，
   暂时移除 selective speculative execution、cache-conditional routing 和复杂 quality delta，
   把论文切口收束到资源受限场景下的专家权重迁移优化。

最终建议的近期实验应从“做复杂 action predictor”改为：

- 建立模型规模和硬件带宽矩阵，先证明哪些配置是 expert-loading/PCIe-bound。
- 在这些配置上跑 token/layer expert access trace。
- 用 simulator 比较 on-demand、LRU、static-hot、Belady-oracle、route-window prefetch 和
  workload-aware replacement。
- 只有当 oracle 与简单 route-window 策略明显优于 LRU，才继续训练 cost-aware policy。

## 2026-04-27 判断 Mixtral offload 在当前平台上的可行性

### 操作

根据用户提问“使用 offload 可以使它跑起来吗”，检查了当前平台资源：

- CPU 内存：125GiB，总可用约 118GiB。
- GPU：2 张 NVIDIA GeForce RTX 3090 Ti，每张 24564MiB。
- GPU driver：535.261.03。
- 磁盘剩余：约 361GiB。
- 当前 base Python 仍缺少 `torch` 和 `accelerate`。

### 操作原理

Mixtral-8x7B 的 BF16/FP16 权重约 90-100GB，不能完整常驻 2x24GB GPU。若使用 CPU offload，权重容量层面有机会放入 CPU 内存，但推理是否稳定运行还取决于 PyTorch/CUDA 环境、Transformers/Accelerate 的 offload 支持、磁盘吞吐、CPU-GPU PCIe 传输开销，以及是否只做 prefill router trace 而不是高吞吐生成。

### 结论

当前平台“有机会用 offload 跑起来小批量 trace”，但不应默认一定能实现稳定高吞吐推理。它适合作为 Phase 2 的受控尝试，不适合作为唯一不可替代路径；应同时保留 Qwen debug trace、OLMoE public trace 和 simulator/HRM scale-out 作为备选。

### 操作原理

Phase 0 不运行模型 forward，只读取 `config.json` 做 HRM-style 估算：

1. 从 config 读取层数、expert 数、top-k、hidden size、MoE intermediate size 和 shared expert size。
2. 按 SwiGLU FFN 结构估算单个 routed expert 参数量：

```text
expert_params = 3 * hidden_size * moe_intermediate_size
```

3. 估算 routed expert 总参数、dense/shared/embedding 参数和 active params per token。
4. 对 BF16、INT8、INT4，不同 GPU memory、KV budget、bandwidth、resident expert fraction、active tokens 做矩阵估算。
5. 在没有真实 route trace 前，用 uniform expected unique experts 近似每层活跃 expert 数。
6. 根据 `transfer_time / (transfer_time + compute_time)` 是否超过 0.30 判断是否 expert-transfer-bound。

### 结论

新增对照说明：

### 操作原理

MoE-Lightning 的 HRM 不是简单判断 expert weights 是否能放进 GPU，或估算 CPU-GPU transfer time。论文第 4.2 节中的 HRM performance model 会对 hardware、model、workload、policy 一起建模，并搜索 policy：

```text
P = (N, micro-batch size, A_g, F_g, r_w, r_c)
```

其中包含 batch size、micro-batch size、attention 是否在 GPU、MoE FFN 是否在 GPU、GPU 常驻 weight 比例、GPU 常驻 KV cache 比例。其 per-layer decode latency 形式是：

```text
T(M,H,W,P) = max(comm_cpu_to_gpu, T_cpu, T_gpu)
```

当前脚本只读取 config，估算参数量、显存占用、all-resident feasibility、粗略 transfer pressure，没有建模 CPU attention、GPU attention、CPU/GPU FFN、KV cache placement、CPU/GPU memory roof，也没有搜索 policy。

### 结论

用户指出的问题成立。当前脚本不应称为完整 HRM 实现，已更正为：

```text
pre-HRM capacity/transfer screen
```

Qwen 的 Phase 0 结论仍可作为容量筛查结论使用，但不能表述为“复现了 MoE-Lightning HRM”。下一步若要贴近 MoE-Lightning，需要新增独立脚本：

```text
scripts/moe_affinity/evaluate_moe_lightning_hrm_policy.py
```

用于评估 policy grid 下的 `T_cpu`、`T_gpu`、`comm_cpu_to_gpu`、`T=max(...)` 和瓶颈标签。

## 2026-04-27 解释 MoE-affinity 含义

### 操作

根据用户提问“moe-affinity 是什么意思”，阅读了当前项目里相关脚本：

- `/root/workspace/scripts/moe_affinity/build_moe_affinity_signature_file.py`
- `/root/workspace/scripts/moe_affinity/simulate_moe_affinity_trace_replay.py`
- `/root/workspace/scripts/runners/run_tp2_moe_affinity_server_formal.sh`

### 操作原理

本项目中的 `moe_affinity` 指一种按 expert signature 相似度重排请求的调度策略。
每个请求先被表示成 sparse expert signature，即“这个请求预计会在各层访问哪些专家以及权重多大”；
调度时在 waiting queue 的小候选窗口内，优先选择和当前 batch 已选请求 expert signature 更相似的请求。

### 结论

`MoE-affinity` 可以理解为“MoE 专家亲和性/专家访问相似性”。它的目的不是改变 MoE router，
而是把专家访问模式相近的请求尽量放在一起执行，从而提高专家 cache/locality 的潜在收益。
