## WREC：解释 Phase 0 未阻塞项的具体做法

### 时间

- 2026-04-28 UTC

### 操作

根据用户对以下事项的疑问，重新对照实验计划并解释其具体执行方式：

```text
INT8/INT4 sweep
带宽 sweep heatmap
Mixtral-8x22B/DBRX/DeepSeek scale-out
真实 route trace
```

参考文件：

- `/root/workspace/notes/wrec_experiment_plan_20260427.md`
- `/root/workspace/scripts/moe_affinity/evaluate_moe_lightning_hrm_policy.py`
- `/root/workspace/scripts/moe_affinity/estimate_moe_hrm_bottleneck.py`
- `/root/workspace/scripts/runtime/benchmark_hrm_hardware.py`

### 操作原理

这四类工作不是同一层级：

1. `INT8/INT4 sweep` 是在 HRM 估计里改变 dtype bytes，观察权重变小后瓶颈是否还存在。
2. `带宽 sweep heatmap` 是在 HRM 估计里改变 CPU-GPU bandwidth，观察传输瓶颈对带宽的敏感性，并画成热力图。
3. `Mixtral-8x22B/DBRX/DeepSeek scale-out` 是读取更大 MoE 的 config/profile，不下载全量权重，用 HRM 外推更大模型是否更容易 transfer-bound。
4. `真实 route trace` 是实际加载模型、跑 prompt forward，从 router 输出中提取每层每 token 的 selected experts；这是 Phase 2，不是 Phase 0。

### 结论

这些工作具体做法如下：

#### INT8/INT4 sweep

目的：

- 回答“如果专家权重量化到 INT8/INT4，显存压力和传输瓶颈是否还存在”。

做法：

- 复用 HRM evaluator。
- 对同一模型分别运行 `--dtype bf16`、`--dtype int8`、`--dtype int4`。
- 比较 total weight、可常驻 expert ratio、`cpu-gpu-transfer` policy 数量、best policy。

注意：

- 这是结构级估计，不等价于真实 INT8/INT4 推理。
- 当前 evaluator 中 dtype 主要影响 bytes；真实 INT8/INT4 kernel 的 `p_g`、`p_c` 需要另测。

#### 带宽 sweep heatmap

目的：

- 回答“CPU-GPU 带宽从 8/16/32 GB/s 到实测 6.5 GB/s 时，WREC 的瓶颈和收益是否更明显”。

做法：

- 固定模型和显存预算。
- 多次运行 HRM evaluator，改变 `--cpu-gpu-bandwidth-gbps`。
- 聚合不同 `r_w`、batch size、bandwidth 下的 bottleneck、latency、transfer time。
- 画 heatmap：横轴可用 `r_w` 或 GPU memory，纵轴用 bandwidth，颜色用 transfer latency 或 bottleneck label。

#### scale-out

目的：

- 回答“更大的 MoE 是否更容易进入 expert-transfer-bound，从而支撑论文动机”。

做法：

- 准备 Mixtral-8x22B、DBRX、DeepSeek-MoE/DeepSeek-V2 的 config 或手写 profile。
- 不强制下载全量权重。
- 用相同 HRM evaluator 计算总权重、expert 权重、可常驻比例、transfer bottleneck。
- 输出 scale-out CSV 和图。

注意：

- scale-out 只作为论文动机和趋势图，不能替代 Mixtral-8x7B 的真实实验。
- 如果某些模型 config 字段和当前脚本不兼容，需要给 evaluator 增加 profile adapter。

#### 真实 route trace

目的：

- 用真实模型运行 prompt，替换 HRM 里的 uniform top-k 估计，得到真实 expert access sequence。

做法：

- Phase 2A 先用 Qwen 本地权重做 debug trace，验证 schema 和统计链路。
- Phase 2B 再做 Mixtral offload feasibility probe：先 1 条 request，再 4 条 request。
- 若 probe 成功，再采 Mixtral debug/train/eval trace。
- trace 每行记录 request、layer、token_pos、selected_experts、expert_probs。

注意：

- 这一步必须下载或具备模型权重。
- Mixtral 可能因为 CPU offload 太慢或内存不足而失败，所以必须先 probe，不能直接跑大规模 trace。
- 这一步属于 Phase 2，不阻塞 Phase 0 的完成判定。


2. Qwen 用于验证 trace 采集和 simulator 链路。
3. 公开 MoE trace 或 synthetic trace 用于补充 cache policy 行为。

## WREC：学术实验中模型选择原则与权重下载成本判断

### 时间

- 2026-04-28 UTC

### 操作

根据用户询问“学术界一般都是怎么判断使用哪个模型，下载模型权重很耗时”，对当前 WREC 方案中的模型选择重新归纳为可执行原则。

### 操作原理

学术实验中选择模型通常不是先下载所有候选模型再试，而是先按研究问题做筛选：

1. 研究问题匹配：模型必须能暴露论文要解决的瓶颈。例如 WREC 要研究 expert weight transfer/cache/offload，因此模型必须在目标硬件上不能轻松 all-resident，并且 HRM/roofline 显示 CPU-GPU expert transfer 占比足够高。
2. 可比性：优先选择已有论文、模型卡、系统论文常用的公开模型，方便和相关工作对齐。
3. 可复现性：config、tokenizer、license、下载方式、硬件要求都要清楚。
4. 可运行性：不能只选理论上漂亮但当前平台无法加载、无法采 trace、无法复现实验的模型。
5. 成本控制：完整权重下载、磁盘占用、加载时间、trace 采集时间都应作为实验成本，不能默认无限制。
6. 阶段分工：通常会设置 debug model、main model、scale-out/motivation model，而不是让一个模型承担全部角色。

因此，下载完整权重应被看作 Phase 2 之后的受控实验动作，而不是 Phase 0/Phase 1 的前置条件。

### 结论

对当前 WREC 来说，模型分工应保持：

```text
Qwen1.5-MoE-A2.7B:
  角色 = debug / tooling model
  用途 = 验证 trace 采集、schema、simulator 输入输出
  不足 = 当前双卡容量下可 all-resident，不适合作为主瓶颈证据

Mixtral-8x7B-Instruct:
  角色 = main bottleneck-evaluation model
  用途 = HRM/config-level 证明 expert-transfer bottleneck；权重可用后做 offload probe 和真实 trace
  前置 = 完整权重下载 + Phase 2B probe 成功

Mixtral-8x22B / DBRX / DeepSeek-MoE:
  角色 = scale-out / motivation model
  用途 = config-only HRM 外推更大模型的 transfer pressure
  前置 = 不要求下载完整权重
```

权重下载决策闭环：

```text
1. 先用 config-only HRM 判断是否值得下载。
2. 如果模型 all-resident feasible 或不是 transfer-bound，不下载完整权重作为主实验。
3. 如果模型 transfer-bound 且 license/磁盘/环境满足，再安排下载。
4. 下载完成后先做 1 条和 4 条 request 的 offload probe。
5. probe 成功才进入真实 trace；probe 失败则停止 Mixtral real-trace 路线。
```

因此，当前不应该因为 Mixtral 权重下载慢而停住主线。合理做法是：

```text
先完成 Qwen Phase 2A event trace + simulator；
并行补 HRM sweep/scale-out；
等 Mixtral 权重下载条件合适时，只把它用于 Phase 2B probe。
```

### 结论

最终 story 应收束为：

1. 过去探索 request scheduling、LTR、MoE-affinity 后发现：只重排请求不能稳定转化为端到端收益，也不能直接证明 expert weight transfer 是瓶颈。
2. 因此论文主线改成先做 HRM-guided bottleneck selection：用 MoE-Lightning-style HRM 判断哪些模型/硬件配置真正 transfer-bound。
3. HRM 结果显示 Qwen 在双卡下可 all-resident，只适合 debug；Mixtral BF16 在当前双 3090 Ti 容量下无法全常驻，并且大量可行 policy 是 CPU-GPU transfer bottleneck，因此成为主瓶颈模型。
4. 选定模型后，不直接设计复杂 predictor，而是先采 token/layer router event trace，得到真实 selected experts 序列。
5. 有了 trace 后，实现 expert cache/offload replay simulator，把 expert miss、transfer bytes、stall proxy、prefetch waste 统一进一个可复现实验环境。
6. 在 simulator 中实现 on-demand、LRU、static-hot、Belady oracle、route-window prefetch，先回答“这个 workload 的 locality 是否真的有可利用上界”。
7. 如果 oracle gap 存在，再提出 WREC：workload-aware route-window expert cache/prefetch，用未来窗口使用概率、routed token workload、miss stall 和 transfer cost 共同决定 keep/evict/prefetch。
8. 最后用 ablation 和 sensitivity 证明收益来自 workload-aware 与 transfer-aware，而不是偶然 cache budget 或窗口大小。

一句话版本：

```text
WREC is a trace-driven expert cache/offload evaluation framework that first uses HRM to select genuinely transfer-bound MoE settings, then replays token/layer expert access traces to compare cache and prefetch policies, and finally uses workload-aware transfer-cost scoring to reduce expert miss stall without changing model routing.
```

最终不应把 vLLM request scheduling、LTR-lite、MoE-affinity request sorting 作为主 story，只应在背景中说它们促成了问题收缩：

```text
早期 request-level scheduling 说明 expert locality 存在，但也暴露了单纯重排请求不足以稳定产生系统收益，因此本文转向更直接的 expert cache/offload 层。
```

看上去不靠谱或当前不宜作为主线的部分：

```text
  - 先证明瓶颈存在；

  - 再证明 oracle 有上界收益；
  - 再证明 WREC 能接近 oracle 且优于 baseline；

```

建议论文主线从开题报告原来的“请求调度、显存准入、专家亲和三线协同”收缩为：

```text
HRM-guided bottleneck selection + trace-driven expert cache/offload optimization
```

## WREC：调整 experiment plan，避免线性打分作为单薄主创新

### 操作

主要调整内容：

1. 在文件开头新增论文级 story：

```text
HRM bottleneck selector
  -> router event tracer
  -> expert working-set / locality analysis
  -> expert cache/offload replay simulator
  -> baseline + oracle gap
  -> WREC workload-aware cache/prefetch policy
  -> ablation / sensitivity
```

2. 明确不再把“线性打分公式”本身作为主创新，而是把 WREC 定义为可证伪的 trace-driven cache/offload evaluation framework。

3. 明确降级内容：

   - 不做 speculative execution。
   - 不改变 router。
   - 不以完整 vLLM runtime 改造作为第一版完成门槛。
   - 不把 `U = alpha * length + beta * affinity` 作为主方法。

4. 在 Phase 3 中重新定义 baseline 与 oracle：
   - online baseline：真实系统可实现，不偷看未来。
   - oracle upper bound：偷看未来，只用于上界，不作为部署策略。
   - WREC candidate：本文方法。

5. 在 Phase 4 中将 WREC 分成三层：
   - WREC-H：可解释启发式，必须实现。
   - WREC-C：带 capacity / queue depth / overlap budget 的 constrained planner，若 WREC-H 太弱则实现。
   - WREC-L：learned cost ranker，可选，只有 trace 和收益稳定后才考虑。

6. 在 Phase 5 中补充：
   - WREC-H vs WREC-C 对比。
   - oracle leakage 检查。
   - online policy 不允许读取 eval future window。

7. 在 Phase 8 和最终成功标准中补充：
   - 图中必须区分 online baseline 与 oracle upper bound。
   - 必须报告 oracle gap capture ratio。
   - WREC 必须在至少 3 个 cache budget 下优于 static_hot 和 online route-window。
   - 若只有 oracle 有收益、online policy 无收益，则结论固定为“离线可利用 locality 存在，但当前在线信号不足”。

### 操作原理

这次调整的核心是把实验计划从“一个打分函数”升级为“一个可证伪的系统实验闭环”：

```text
不是证明 score 公式漂亮；
而是证明瓶颈存在、trace 有 locality、oracle 有上界、baseline 可比较、WREC 能捕获部分 oracle gap。
```

这样可以避免最终论文被质疑为“杂糅的线性启发式”，同时保留开题报告中有价值的 expert working set、cache/offload、状态约束和资源受限 MoE 主线。

### 结论

调整后的 experiment plan 更适合论文展示：

1. 主贡献不再是 EALR 式线性拼接。
2. Phase 3 的 simulator 与 baseline/oracle 成为核心工作量。
3. Phase 4 的 WREC 至少有 WREC-H；若 H 太弱但 oracle gap 明显，则升级到 WREC-C。
