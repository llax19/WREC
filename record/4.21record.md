16. 对 `GPU 0` 异常显存占用做了更深入诊断：
    - `nvidia-smi -q -d MEMORY,COMPUTE,PIDS` 显示 `GPU 0` 上存在一个约 `21880 MiB` 的计算上下文

    - `nvidia-smi --query-compute-apps` 将该进程名显示为
      `[Not Found]`
    这强烈说明问题更像是陈旧或孤儿态的 CUDA 上下文，而不是一个正常存活的进程。
17. 尝试执行 `nvidia-smi --gpu-reset -i 0`，但 reset 失败并返回 `Unknown Error`，因此当前回收 `GPU 0` 最稳妥的方式看起来还是整机重启。
18. 检查了本地 `vLLM` 的清理路径，以判断是否是 smoke test 脚本自己忘记释放 GPU 显存：
    - [vllm_smoke_load.py](/root/workspace/scripts/vllm_smoke_load.py)
      原本只是 `LLM(...) -> generate(...) -> print(...)`，没有显式 shutdown 调用
    - 但已安装的 `vLLM` 代码在启动失败时本身就会尝试清理失败的 engine-core 子进程
    - 进程管理器使用 terminate/kill 方式关停，因此当前观察到的显存残留更像是底层 CUDA/driver 清理失败，而不是简单少写了一个 Python `del`
19. 加固了
    [vllm_smoke_load.py](/root/workspace/scripts/vllm_smoke_load.py)
    ，加入了尽力而为的显式 shutdown 路径，以及 `gc.collect()` 和
    `torch.cuda.empty_cache()` 到 `finally` 中，这样无论是正常运行还是后续异常，都会尽量减少 Python 侧残留。
20. 新增了
