我们这次敲定的新实验 idea 可以概括成：

**WREC Selective Expert Prefetch**

核心不再是“finite-slot expert cache 常驻 GPU 后提升推理”，而是：

> 在固定 GPU staging buffer 预算下，利用 attention 计算窗口，提前预取 WREC 预测会被下一次 MoE forward 使用的 experts，从而减少或隐藏 expert offload/UVA 访问开销。

具体机制：

1. **GPU 侧预分配固定上限的 staging slots**
   ```text
   max_slots = min(num_experts, top_k * max_num_batched_tokens)
   ```

2. **每层 attention 开始时异步 prefetch**
   根据 WREC score 预测本层可能用到的 experts，把它们从 CPU pinned memory 预取到 GPU staging slots。

3. **attention 后运行真实 gate**
   得到真实 top-k experts，计算：
   ```text
   U = 本次 forward 真实 unique experts 数
   H = 预取命中的 experts 数
   M = missing experts 数 = U - H
   ```

4. **forward 前处理 missing experts**
   如果剩余 slots 能容纳 missing experts，则同步补齐后走 GPU staging buffer forward。  
   如果放不下，则 fallback 到原 UVA/offload 路径。

5. **active slots 动态变化**
   每次 forward 可以只启用部分 slots：
   ```text
   active_slots = alpha * top_k * num_tokens_this_forward
   ```
   但这只减少本轮传输和 waste，不减少已分配的显存。

新的 story 应该这样讲：

> vLLM UVA offload 是低 HBM 占用的 conservative baseline，但在 prefill-heavy workload 中，attention 提供了隐藏 expert transfer 的机会。整层 expert prefetch 没有 miss，但传输大量无用 experts。WREC 的目标是在固定 staging buffer 和 transfer budget 下，选择更可能被访问的 experts，降低 miss 和无效传输。

关键实验指标：

```text
max_slots
KV cache loss
U distribution
prefetch hit rate
missing experts M
sync补拷贝 bytes/time
waste transfer
TTFT
prefill throughput
decode throughput
fallback rate
```

目前最重要的判断：

- `2k` slots 不适合作为 prefill 主方案。现有 trace 下 top-k=4，chunk=4 时 `U>8` 比例约 95.7%。
- `2k` 可以作为 decode/小 microbatch case。
- prefill 主实验更应该扫：
  ```text
  max_slots = 8, 16, 24, 32
  ```
  或者用：
  ```text
  max_slots = k * max_num_batched_tokens
  ```

最终定位要克制：

> WREC 不是免费提升 serving，也不是减少 KV cache 压力；它是在明确牺牲一部分 HBM/KV cache 预算后，尝试用更聪明的 expert prefetch 换取 prefill 阶段收益。

## 2026-05-15 shared slots 语义修正

操作：
- 将 WREC finite slots 从每层独立分配改成共享 staging pool。
- 删除误导性的 `_slot_capacity_for_layer` 命名，改为 `_max_shared_slots`。
- 每个 MoE layer 仍保留自己的 CPU backing 和 expert_map，但相同 shape/dtype/device 的层挂到同一组 slot 参数上。

原理：
- Transformer 层按顺序执行，同一时刻只需要当前 MoE 层的专家权重占用 staging slots。
- 当某层 prefetch/forward 前 claim shared pool 时，会等待已有异步拷贝完成，并清空上一个 owner layer 的 resident/expert_map 状态。
- 因此 slots 是跨 layer 复用的；每次 MoE forward 前如果 attention prefetch 没有命中，会同步补齐当前 layer 的真实 experts。

结论：
- 当前实现符合“所有 layer 共用一份 slots”的实验语义；严格地说，代码按 expert 权重 shape/dtype/device 建 pool，常规同构 MoE 层会共享同一份 slots。
