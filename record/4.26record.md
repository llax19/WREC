## 2026-04-26 Qwen router trace 采集脚本准备

### 操作

新增脚本：

- `/root/workspace/scripts/build_qwen_router_trace_signatures.py`

脚本用途：

1. 读取 `data/prompts/eval_requests.jsonl` 这类请求 manifest。
2. 用 Transformers 加载本地 `qwen1.5-MoE-A2.7B`。
3. 对每条 prompt 做离线 prefill forward，并启用 `output_router_logits=True`。
4. 从每层 router 概率中取 top-k experts，聚合成 request-level sparse signature。
5. 输出现有 `moe_affinity` scheduler 与 `simulate_moe_affinity_trace_replay.py` 都能读取的 JSONL schema。

完成的轻量验证：

- `python -m py_compile scripts/build_qwen_router_trace_signatures.py`
- `python scripts/build_qwen_router_trace_signatures.py --help`

### 操作原理

本地模型 config 显示：

- `model_type=qwen2_moe`
- `num_hidden_layers=24`
- `num_experts=60`
- `num_experts_per_tok=4`
- `decoder_sparse_step=1`

当前 Transformers 环境中的 `Qwen2MoeTopKRouter` 会输出每 token 对 60 个 expert 的
router probability；`Qwen2MoePreTrainedModel` 已注册：

```text
router_logits: OutputRecorder(Qwen2MoeTopKRouter, index=0)
```

因此离线 forward 时打开 `output_router_logits=True` 就可以直接拿到每层 router
输出，不需要先侵入 vLLM forward 路径。脚本把每层每 token 的 top-k expert 概率累加，
默认按 token 数归一化，再每层保留 top-16 experts，得到和现有 sidecar 一致的：

```json
{"request_id": "...", "num_layers": 24, "num_experts": 60, "experts": [[layer, expert, weight], ...]}
```

### 结论

下一步可以先用该脚本采一小批真实 Qwen router signature，例如前 24 条请求；如果能正常生成，
立刻复用现有 replay simulator 检查 Qwen trace 上 `affinity` 是否相对 `fcfs/length`
提高 batch cosine、降低 unique experts 和 cache miss proxy。

### 操作原理

真实 router trace 的作用是替换此前的 proxy signature。离线 replay 用它验证是否存在
可调度的 expert locality；服务端 smoke 则验证这种 locality 是否能转化为端到端 latency。

topK 是当前最有效的保守旋钮。`VLLM_MOE_AFFINITY_WEIGHT` 只要大于 0，在同一个 bucket
里就是按 affinity 单调排序，因此调小 weight 并不会真正改变候选内部顺序；要减少重排，
应优先调小 `VLLM_MOE_AFFINITY_TOPK` 或缩小 bucket。

### 结论

真实 Qwen router signature 已经证明比 proxy signature 更有意义：

1. 离线 replay 中，Affinity 明确优于 Length 的 expert locality proxy。
2. 服务端 latency 中，topK=16 明显优于 FCFS，但三轮均值仍未稳定超过 Length-only 的
   P95/P99 TTFT。
3. topK=8 太保守，基本退化成 Length-only；topK=16 有时能略优于 Length-only 单轮，
   但稳定性还不够。

当前不应直接扩大服务端 latency 到 72/240 作为论文结论。更合理的下一步是改进
`MoEAffinityPolicy.dynamic_sort_key`：在同 bucket 内先保留更强的 length/base-score
保护，再把 affinity 作为次级 tie-breaker 或只在长度差很小的候选之间生效。这样有望保留
真实 router locality 的 miss reduction，同时避免 P95/P99 TTFT 被局部重排抬高。

## 2026-04-26 路线重新讨论：route trace 排序是否还值得作为主线

### 操作

根据用户提出的问题，重新评估 `route trace -> affinity 排序 -> latency 改善` 这条路线的必要性。

### 操作原理

判断重点不再是“能不能继续调参”，而是看这条路线是否满足论文主线所需的两个条件：

1. 即使使用真实 router trace 这种 oracle-ish 信号，服务端 latency 是否能稳定超过
   `Length-only`。
2. 该信号是否能在线获得；真实 router trace 需要先跑 prefill，天然晚于 waiting queue
   调度决策，因此如果没有 predictor，它更像诊断/上界，而不是可直接部署的调度信号。

### 结论

当前形式的 route-trace affinity 不适合作为下一阶段 latency 主线：

- 它在离线 replay 中证明了真实 expert locality 存在。
- 但端到端 high 档三轮没有稳定超过 `Length-only` 的 P95/P99 TTFT。
- 更关键的是，真实 route trace 本身不是调度前可得信号；如果 oracle trace 都不稳定超过
  `Length-only`，继续投入 predictor 训练的优先级应降低。

后续只有在研究问题改写为 expert cache/prefetch、expert offload、或 MoE 权重迁移成本优化时，
route trace 才仍然值得继续作为核心信号。若目标仍是当前 vLLM/Qwen 服务端 request scheduling
latency，则应把它降级为 ablation/negative finding，主线转向更稳的 length-first scheduler
或其他能直接影响 TTFT/TPOT 的机制。

## 2026-04-26 论文主线转向 expert cache / expert prefetch

### 操作

根据论文题目“面向资源受限环境的混合专家大模型多维资源协同推理优化方法”，重新定位研究主线：

- 不再把 request sorting 作为核心贡献。
- 将 route trace / expert affinity 从“请求排序信号”改定位为“专家缓存、预取、offload 决策信号”。
- 把资源受限场景下的 GPU 显存、CPU offload、专家权重迁移、KV cache 与请求调度统一到多维资源协同优化框架中。

### 操作原理

请求排序只改变 waiting queue 顺序，能影响 TTFT，但没有直接解决 MoE 在资源受限环境中的核心问题：

1. 专家权重数量多，活跃专家稀疏，GPU 显存无法长期常驻所有专家。
2. CPU/GPU 之间专家迁移和缓存命中率会影响 TPOT、wall time 和吞吐。
3. route locality 的真正价值不是“让相似请求排一起”，而是提前知道未来短窗口会访问哪些 expert，
   从而指导 cache eviction、prefetch 和 offload placement。

因此真实 route trace 的实验结果不应被视为失败，而应重新解释：

- 离线 replay 已证明 expert locality 存在。
- request sorting 不能稳定把 locality 转化为 latency 收益。
- 这说明 locality 信号需要接入 expert cache/prefetch 机制，而不是只接入 waiting queue 排序。

### 结论

下一阶段应转向 expert cache / expert prefetch 方向，形成更贴合论文题目的主线：

1. 建立 expert access trace 与 cache simulator，衡量不同 cache/offload 策略下的 expert miss、迁移量和理论收益。
2. 设计 locality-aware expert cache policy，例如 LRU、LFU、Belady-oracle、route-window prefetch、length-aware prefetch。
3. 在可控模拟中证明 route trace/predictor 能降低 expert miss 与 CPU-GPU 迁移成本。
4. 若时间允许，再把该策略接到 vLLM 或一个轻量 MoE inference harness 中做端到端验证。

当前 request sorting 结果可保留为反证/边界实验：单纯改变请求顺序不足以稳定超过 Length-only；
真正的 MoE 资源优化应作用在 expert cache/prefetch/offload 层。

### 操作原理

1. MoE 的核心资源压力来自 expert 权重无法全部常驻 GPU，而不是 waiting queue 顺序本身。
2. route trace 的价值应体现在 expert cache、prefetch、eviction 和 offload stall 避免上。
3. 预测目标应从 `expert id` 改成 `cache/prefetch action`，训练目标应从 prediction accuracy

```text
miss penalty + transfer cost + stall cost + quality penalty
```

- 它解决什么系统问题；

- 它留下的空白如何转化为我们的创新点。

### 结论

后续主线建议正式转为：

> Cost-aware route-trace-driven expert cache manager

也就是：用历史 route trace、cross-layer gate signal、router entropy/margin、prompt/domain
信息和当前 GPU cache bitmap，直接预测 `prefetch / keep / evict / skip / speculative alternative`
等动作，并用系统代价而非 expert prediction accuracy 训练和评价。

最小落地路线：

1. 新增 token/layer 级 Qwen router event trace 采集脚本，把当前 request-level signature
   升级为可模拟 cache action 的 event trace。
2. 新增 expert cache/offload simulator，比较 on-demand、LRU、static-hot、MoE-Infinity-style、
   Fate-style、DALI-style、oracle 和 cost-aware policy。
3. 若 simulator 显示 route-window oracle 明显优于 LRU，再训练轻量 action predictor。
4. 最后只在小规模真实 forward 或 lightweight harness 中验证 selective speculative execution
   的质量边界，不强行把完整 expert cache engine 一步接入 vLLM。

该方向比继续做单纯 request sorting 更贴合论文标题，也能自然继承当前已经完成的 Qwen route
trace 与 OLMoE route locality replay 结果。
