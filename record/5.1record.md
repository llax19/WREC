- 如果只是做开发调试，可以只用 debug trace 或先跑 eval trace 看趋势。
- 如果要做公平版 `static_hot` / WREC 主表，仍然需要一个和 eval 分开的参考 trace；这就是计划里 `train trace` 的用途。

## static_hot 方法说明

### 本次操作

- 对齐 `static_hot` 在实验计划和 simulator 脚本里的定义，回答它到底是什么方法。

### 操作原理

- 计划文档把 `static_hot` 定义为：根据参考 trace 里的 expert frequency，固定缓存每层最热门的 experts。
- 对应实现见 [simulate_expert_cache_offload.py](/root/workspace/scripts/moe_affinity/simulate_expert_cache_offload.py:135)：先统计每层各个 expert 的出现次数，再选出 top-k。
- 初始化缓存时，`static_hot` 会直接把这组 top-k experts 预放进每层 cache，见 [simulate_expert_cache_offload.py](/root/workspace/scripts/moe_affinity/simulate_expert_cache_offload.py:160)。
- 运行过程中它不会像 LRU 那样动态替换缓存内容；如果访问到不在这组固定热门集合里的 expert，就按 miss 处理并发生 demand load，见 [simulate_expert_cache_offload.py](/root/workspace/scripts/moe_affinity/simulate_expert_cache_offload.py:327)。

### 结论

- `static_hot` 是一个离线 popularity baseline，不是在线自适应策略。
- 它回答的问题是：如果只靠“全局最热门 experts”这件事，能不能已经解释大部分 locality 收益。
- 做公平实验时，`static_hot` 的热门统计必须来自和评测 trace 分开的参考 trace；如果直接用 eval trace 自己统计，会有信息泄漏。

## trace 命令与 simulator 逻辑澄清

### 本次操作

- 核对 `build_moe_router_event_trace.py` 和 `simulate_expert_cache_offload.py`，澄清 trace 采集命令本身是否包含多种替换策略，以及 simulator 的工作流程。

### 操作原理

- `build_moe_router_event_trace.py` 的职责只有一件事：加载模型、跑请求、输出每个 token 在每层选中的 experts，也就是 router trace。
- 该脚本会调用模型 forward，并要求返回 `router_logits`，再对每层每个 token 取 top-k expert，写成 jsonl 事件流；它不包含 `lru`、`static_hot`、`belady_oracle` 等替换策略逻辑。
- 真正的替换策略发生在 `simulate_expert_cache_offload.py`：
  - 先把 trace 展开成逐 expert 的访问序列；
  - 按 layer 维护 resident cache；
  - 根据 policy 决定 hit/miss 后是否换入、如何 eviction、是否预取；
  - 最后按 `expert_bytes` 和 `bandwidth_gbps` 计算 transfer bytes 与 stall proxy。
- 当前 simulator 的 stall 定义是 demand transfer bytes 除以带宽，不建模 compute overlap、并发调度或真实 runtime pipeline。

### 结论

- `build_moe_router_event_trace.py` 是 trace 采集脚本，不会同时跑多种替换方案。
- 多种替换方案是在后续 replay simulator 中对同一份 trace 分别回放比较。
- 当前 simulator 本质上是一个“基于真实 router trace 的离线 cache/offload replay”，输出的是 hit/miss、transfer、stall proxy 和 oracle gap，而不是端到端真实运行时延。

## router event 定义澄清

### 本次操作

- 对齐 trace 采集脚本与 simulator 的输入结构，回答“一条 router event”在当前实验里具体指什么。

### 操作原理

- 当前 trace 采集脚本对每个 `request_id`、每个 `layer`、每个 `token_pos` 写出一条 json 事件，其中包含该 token 在该层被 router 选中的 `selected_experts` 列表及对应概率。
- 因此，一条 `router event` 表示的是“一次 token 在某一层上的 routing 决策”，而不是“单个 expert 的一次访问”。
- 在 Mixtral 这类 top-k 路由模型里，一条 event 里可以同时包含多个 expert；后续 simulator 会再把它展开成多个逐 expert 的 `ExpertRef` 访问记录。

### 结论

- `router event` 的粒度是 `(request, layer, token)`。
- `expert ref` 的粒度才是 `(request, layer, token, expert)`。
- 所以对 Mixtral top-2 路由，`19 tokens * 32 layers = 608 router events`，但会展开成 `19 * 32 * 2 = 1216 expert refs`。

## top-2 routing 定义澄清

### 本次操作

- 解释 `top-2 routing` 在 Mixtral MoE 路由中的具体含义。

### 操作原理

- `top-2 routing` 指的是：对某个 token，在某一层 router 给所有 experts 打分后，只保留得分最高的两个 experts。
- 当前 trace 脚本会把 router logits 转成概率，然后按 top-k 选中 experts；其中 `k` 默认来自模型配置里的 `num_experts_per_tok`。
- 对 Mixtral，这个值是 2，所以每个 token 在每个 MoE layer 会被路由到 2 个 experts。

### 结论

- `top-2 routing` 的“2”表示每个 token 每层只激活两个 experts，不是总共有两个 experts。
- 因此一条 Mixtral router event 里通常会有两个 `selected_experts`，后续 simulator 也会把它展开成两次 expert reference。
