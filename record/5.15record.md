## Hit-rate 差异解释

- 操作：对比 `logs/processed/qwen_uva_reuse_smoke_20260515T/wrec_residency_stats.jsonl`、`results/wrec/runtime_selective_prefetch_smoke_short_20260515T/wrec_residency_stats.jsonl` 和 `results/wrec/runtime_lazy_init_smoke_20260515T/wrec_residency_stats.jsonl`。
- 结果：旧 UVA reuse smoke 为 `U=908,H=63,M=845`，`sum(H)/sum(U)=0.0694`，row-average `0.0456`，max `0.3333`；新 lazy-init smoke 为 `U=908,H=166,M=742`，`sum(H)/sum(U)=0.1828`，row-average `0.1170`，max `0.6923`。
- 结论：新 lazy-init smoke 和上一轮 runner short smoke 的 stats 完全一致，因此 hit-rate 变高不是 lazy init 本身带来的。主要差异来自 sidecar 配置：旧手工 smoke 很可能使用 sidecar 默认 `total_slots=64`，新 runner 使用 `SIDECAR_TOTAL_SLOTS=192`，使 desired/admit expert 数更多；同时 2 条重复 short prompt 会放大第二条请求的 request-local 命中。该现象可解释为机制上更宽松的 shadow cache 预算提高了 prefetch overlap，但不能作为性能收益或统计稳定结论。

## Ranked expert prefetch 实现

- 操作：把 prefetch 候选从 `would_admit` desired set 改为 sidecar 每层全 expert 排名。sidecar 新增 `/rankings`，每次 `/event` 返回当前层和下一层 ranking；vLLM sidecar client 启动时拉取初始 ranking；expert residency manager 优先用 `ranked_experts_by_layer[layer][:active_slots]`，没有 ranking 时 fallback 到旧 desired experts。
- 原理：`active_slots` 由 runtime 根据当前 attention/MoE token 数和 slot capacity 决定，因此候选生成应先按每层 expert score 排序，再在 runtime 侧截断，避免 sidecar shadow cache budget 决定候选数量。
- 验证：`py_compile` 通过；sidecar HTTP smoke 通过，`/rankings` 返回 24 层、每层 60 experts；事件响应返回当前层和下一层 ranking；WrecSidecarClient 初始拉取 ranking 后 manager 中有 24 层 ranking；fake manager smoke 证明 attention hook 会按 ranking top-active slots 预取。
- 部署：通过 `deploy_vllm_python_patch.py` 同步到 `vllm_moe`，manifest 为 `results/wrec/runtime_patch_deploy/vllm_python_patch_deploy_20260515T020157Z.json`。

- block-level：block1 `0/96`，block2 `0/96`，block3 `243/358=0.6788`，block4 `358/358=1.0000`。

- 结论：ranked prefetch 机制生效并显著提高 hit ratio，但无条件取满 `active_slots=32` 太激进，短请求下 H2D transfer 开销明显；后续需要 threshold 或 per-layer prefetch budget。
