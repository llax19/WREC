## Ranked expert prefetch 实现

- 操作：把 prefetch 候选从 `would_admit` desired set 改为 sidecar 每层全 expert 排名。sidecar 新增 `/rankings`，每次 `/event` 返回当前层和下一层 ranking；vLLM sidecar client 启动时拉取初始 ranking；expert residency manager 优先用 `ranked_experts_by_layer[layer][:active_slots]`，没有 ranking 时 fallback 到旧 desired experts。

## mbt 默认值和 slot 开销估算

- 操作：检查当前安装的 vLLM `arg_utils.py` 与 `SchedulerConfig`，并用 sidecar metrics 中的 `expert_bytes=17301504` 估算 Qwen WREC slot 显存。
- 原理：vLLM OpenAI API server 在非 H100/H200 GPU 上默认 `max_num_batched_tokens=2048`；若用户显式设置 `max_num_seqs=1` 和 `max_model_len=1280` 且不传 `--max-num-batched-tokens`，最终会裁剪为 `min(2048, 1*1280)=1280`。当前 runner 不是这个默认路径，它总是通过 case 传入 `mbt`。
- 估算：单个 Qwen expert slot 包含 `w13+w2`，约 `16.5 MiB`。slot `16/24/32/60/64/192` 分别约 `264/396/528/990/1056/3168 MiB`。
- 结论：不显式控制 mbt 时，当前 runner 配置下等价于把 chunk 上限放到 `1280`，会使 long prefill 不再被小 chunk 切碎；但 finite-slot 的静态 GPU slot pool 开销主要由 `slot_capacity` 决定，slot=32 仍约 `528 MiB`，不会因为 mbt 变成 1280 自动涨到 1280 个 slots。mbt 变大主要影响每个 MoE forward 的 token 数、unique expert 需求和 copy/prefetch 压力。

## long prefill expert working set 判断

## 长 prefill 延迟改进方向判断

- 操作：结合 `no_wrec/state_only/finite_slot` 三组 long prefill 对照，判断是否应通过提高并发缓解当前延迟。
- 原理：若主要瓶颈来自单请求关键路径上的 sidecar per-event CPU/HTTP 开销，提高并发只会帮助吞吐，不会降低单请求 TTFT；甚至可能放大 tail latency。
- 结论：当前首要改进不是提并发，而是先减少每请求关键路径工作量。优先级应为：1) 合并/批量化 routed expert export 与 sidecar `/event` 调用；2) 不要每个 router event 都回传 ranking，改成每 layer/chunk 或预拉全层 ranking；3) 将 sidecar 轻量逻辑迁入进程内或共享内存路径，去掉 Python HTTP 往返；4) 仅在 sidecar 路径明显变轻后，再评估提高 `MAX_CONCURRENCY` 对吞吐的收益。对当前 long prefill，提升并发预计不会把 `~95s` TTFT 降回接近 `4s` baseline。

## sidecar 批量提交优化第 1 步

- 操作：修改 `scripts/wrec/runtime/runtime_sidecar.py` 与 `external/vllm-0.19.0/vllm/v1/core/sched/wrec_sidecar_client.py`，让 `/event` 同时支持单 event 和 `{"events": [...]}` 批量提交；vLLM client 默认按 token 聚合同一 token 的整层 router events 后再提交。随后用 `deploy_vllm_python_patch.py` 将补丁部署到 `vllm_moe` 环境。
- 原理：当前长 prefill 的主要放大项是 `27096` 次 router event 对应 `27096` 次 Python JSON/HTTP/handler 往返。先把提交粒度从 per-event 改成 per-token batch，可在不改策略语义的前提下压缩请求次数。sidecar 端同时把批处理中的 ranking 改为“整批尾部算一次”，避免只是把多次 HTTP 变成一次 HTTP、却仍在服务端重复做每 event ranking。
- 困难：1) 当前 sandbox 无法直接工作，读取和校验命令需要改走提权执行；2) 批处理初版若直接复用 `process_event()`，会在一批内重复返回 metrics 和 ranking，收益会被内部重复计算吃掉。处理方式是拆出 `_process_event(..., include_rankings, include_metrics)`，批量路径只保留状态推进和决策，尾部统一生成 `ranked_experts_by_layer` 与 metrics。
- 验证：对 `results/wrec/runtime_vllm/vllm_routed_experts_qwen_events_20260505.jsonl` 前 `48` 个 events 做单 event 与按 token 分组对照，`decisions` 与 `expert_refs/router_events/shadow_hits/shadow_misses/would_*` 全量一致；两个修改文件也都通过 `py_compile`。
- 结论：第 1 步已经落地到代码与当前 `vllm_moe` 运行环境，当前默认行为已从 per-event HTTP 提交切到 per-token batch。下一步应直接测同一条 long request 的 TTFT 与 sidecar timing，确认 `online_loop` 与 wall time 是否按预期下降，再继续做“不要每个 /event 都返回 ranking”。

- 结论：第 2 条已经验证成立，而且收益很干净。现在 sidecar 总体瓶颈已经从 ranking 路径转回主循环决策路径；若继续优化，应优先看第 3 条，把 sidecar 逻辑从 Python HTTP 进程外移到进程内。

## sidecar 进程内化改造规划

- 操作：梳理把 `runtime_sidecar.py` 的核心逻辑迁入 vLLM 进程内的最小改造范围，明确后续实施应改哪些文件。
- 原理：当前 ranking 路径已基本压掉，但 `online_loop_seconds` 仍约 `62s`，说明主要剩余开销来自 Python HTTP/JSON/handler/object build 以及外部 sidecar 调用边界，而不是 ranking 频率本身。进程内化的目标是保留同一套 WREC shadow/cache/policy 语义，同时去掉 HTTP server/client 协议层。
- 规划：
  - 抽公共 engine：把 `scripts/wrec/runtime/runtime_sidecar.py` 中 `WrecRuntimeSidecarEngine` 与相关数据结构拆到一个纯 Python engine 模块，HTTP handler 只做薄包装。
  - 增加进程内 client：在 `external/vllm-0.19.0/vllm/v1/core/sched/` 下新增本地 engine bridge，替代 `wrec_sidecar_client.py` 的 HTTP submit 路径，直接在 vLLM 进程里初始化 engine 并调用批处理函数。
  - 保留 fail-open：维持当前 sidecar client 的容错语义，engine 初始化失败时自动降级，不阻断 vLLM 主流程。
  - 配置入口：把 `train_trace/model_path/slots/threshold/ranking_refresh_router_events` 等参数改成通过环境变量或 vLLM 内部配置传给进程内 engine，而不是只传给外部 sidecar 进程。
  - runner 分流：`run_qwen_real_inference_main.sh` 需要支持两种模式，`external_sidecar` 与 `inproc_sidecar`；进程内模式下不再启动 `runtime_sidecar.py`，也不再设置 `WREC_SIDECAR_URL`。
  - metrics 复位：当前 `/metrics` 需要替换成进程内 metrics 导出路径，至少要能把最终 engine metrics 写到结果目录，保持与现有实验表兼容。
  - 校验：先做小 trace 单测，确认进程内 engine 与外部 HTTP sidecar 的 `decisions` 和最终 metrics 一致；再复跑 long case 比较 TTFT 与 `online_loop_seconds`。
- 结论：进程内化不是再调参数，而是把“HTTP sidecar + client”变成“vLLM 内直接函数调用同一套 engine”。最小改造核心在三处：抽公共 engine、替换 `wrec_sidecar_client.py` 提交路径、重做 metrics 导出；runner 只做模式切换和结果采集。

## engine miss 路径与 batch 聚合热点评估优化

- 时间：`2026-05-15T10:00:57Z`
- 操作：修改 `scripts/wrec/policy.py`、`scripts/wrec/runtime/runtime_engine.py`、`scripts/wrec/runtime/shadow_runtime.py`。具体包括：1) 给 `policy.admit_or_bypass()` 增加结构化返回 `WrecAdmissionResult`；2) `runtime_engine` 的 miss 路径不再通过 `before=set(resident)` / `after=set(resident)` 反推 admission/eviction，而是直接消费结构化结果；3) `process_events()` 改为流式累积 `decisions`，不再先缓存全部 `responses` 再二次 flatten；4) `shadow_runtime` 同步改为使用结构化结果记账。
- 原理：当前 long prefill 指标显示 `decision_us_per_miss=715us`，远高于 `history_update_us_per_expert_ref=3us` 和 `ranking_us_per_router_event=7.9us`，说明 admission miss 路径才是主热点。原实现每次 miss 都复制整个 `resident` 集合并做差分，复杂度随 cache 容量线性增长；把 admission 结果直接从 policy 返回，可以把这部分额外的 `O(total_slots)` Python 集合工作删掉。`process_events()` 的 response 列表也是纯额外内存和遍历开销，流式累积即可保留同一返回语义。
- 困难：1) 当前命令沙箱缺少 `bwrap`，普通读取和校验命令会直接失败；处理方式是改用提权命令完成代码检查与校验。2) `admit_or_bypass()` 被 `runtime_engine`、`shadow_runtime`、benchmark 共享，不能只改单一调用点；处理方式是把返回值设计成兼容旁路调用的只读 dataclass，老调用点即使暂时不消费结果也不会破坏行为。
- 验证：`python -m py_compile scripts/wrec/policy.py scripts/wrec/runtime/runtime_engine.py scripts/wrec/runtime/shadow_runtime.py` 通过。补充了两个最小内存 smoke check：一组验证 bypass 路径会返回 `admitted=False`，另一组验证满缓存时高分新 expert 会返回 `admitted=True, evicted=(layer, expert)`，结果均符合预期。
- 结果：本轮先完成确定性的热路径删减，没有改 ranking 逻辑，因为现有实测中 ranking 只占很小比例，继续改它不符合当前瓶颈排序。
- 结论：这次改动针对的是 engine 本身最可疑的算法浪费，尤其是 miss 路径的 resident 全量复制。下一步应直接复跑你当前的 long prompt case，对比 `decision_us_per_miss` 和 `online_loop_us_per_expert_ref` 是否下降；如果下降有限，再继续看 `_process_event()` 调用粒度或进一步把 request history/update 路径内联化。
