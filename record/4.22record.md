### 运行过程中的发现

1. `GPU 0` 目前仍不适合承担阶段一运行，因为它依旧显示存在较重的隐藏显存占用，因此当前稳定路径使用的是 `GPU 1`。
2. 最初几次单卡服务启动虽然模型加载成功，但最终仍然失败：
   - 当 `gpu_memory_utilization=0.60` 时，CUDA graph 和 KV cache 初始化后没有留下可用 cache blocks
   - 即使切到 `gpu_memory_utilization=0.75` 并启用 eager mode，问题仍然存在
3. 在切换到更宽松的单实例显存上限后，运行变得稳定：
   - `gpu_memory_utilization=0.90`
   - `max_model_len=512`
   - `--enforce-eager`
4. 一开始本地 API 探测看起来不稳定，是因为 Python `urllib` 请求仍然沿用了环境中的 opener 路径；在把客户端脚本切换为不走代理的 opener 后，`/health` 和 `/v1/completions` 都能从实验脚本中正常访问。

### 单卡跑通的关键原因

这次之所以最终能让模型在单卡上跑起来，不是因为模型突然“变小了”，而是因为把显存预算和运行方式调到了适合当前机器状态的一组组合。

1. 先绕开不可控变量：
   - 默认双卡路径依赖 `GPU 0` 和 `GPU 1` 同时可用
   - 但当前 `GPU 0` 存在隐藏显存占用，无法稳定参与 tensor parallel
   - 因此阶段一不再强行追求 `TP=2`，而是直接绑定 `GPU 1`
2. 控制上下文长度，降低 KV cache 压力：
   - 早期尝试使用 `max_model_len=1024`
   - 权重虽然能加载，但后续 KV cache 初始化时显存不够
   - 将 `max_model_len` 压到 `512` 后，单请求的缓存预算明显下降
3. 关闭 CUDA graph 路径，减少额外显存开销：
   - 加上 `--enforce-eager` 后，不再走 CUDA graph / torch.compile 路径
   - 这一步不是为了提速，而是为了减少启动阶段的额外图编译和图缓存开销
4. 保留 CPU offload，给 GPU 留空间：
   - `cpu_offload_gb=8` 让一部分参数留在 CPU/UVA 路径
   - 对当前单卡阶段一验证来说，这个取舍比纯 GPU 常驻更稳
5. 关键不是一味“降低”显存利用率，而是给 KV cache 留出可分配空间：
   - 一开始直觉上会觉得 OOM 就该继续下调 `gpu_memory_utilization`
   - 但这次实际踩到的问题不是“权重放不下”，而是“KV cache block 没有可用空间”
   - 因此从 `0.60` 提到 `0.90` 反而是正确方向

最终稳定可用的单卡参数组合是：

- `CUDA_VISIBLE_DEVICES=1`
- `tensor_parallel_size=1`
- `gpu_memory_utilization=0.90`
- `cpu_offload_gb=8`
- `max_model_len=512`
- `--enforce-eager`

### 踩坑总结

这次阶段一里真正踩过、并且后面很容易重复踩的坑如下。

1. 看起来像 OOM，但并不总是“权重装不下”：
   - 在 `20260422T062747Z_vllm_server.log` 中，模型权重已经完成加载
   - 真正报错点出现在 KV cache 初始化阶段，日志里明确有
     `Available KV cache memory: -5.18 GiB`
     和
     `No available memory for the cache blocks`
   - 这说明要区分“权重加载失败”和“缓存初始化失败”两种问题
2. `gpu_memory_utilization` 不是越小越安全：
   - `0.60` 时失败
   - `0.75` 时仍失败，日志里还有
     `Available KV cache memory: -1.48 GiB`
   - 直到切到 `0.90` 才真正为 KV cache 留出正向可用空间
   - 这是这次最容易误判的一点
3. `GPU 0` 的隐藏占用会把排查方向带偏：
   - 表面上看像是“双卡脚本不稳”
   - 实际上是 `GPU 0` 本身就不干净，导致 `TP=2` 根本不具备稳定前提
   - 如果不先接受“阶段一先单卡”的现实，会在双卡上浪费很多时间
4. 服务日志显示启动成功，不代表客户端一定能立即打通：
   - 当时服务端已经能响应 `/health`
   - 但 Python `urllib` 客户端仍然出现过 `RemoteDisconnected`
   - 根因是本地请求路径受到了环境 opener / 代理链路影响
   - 解决方式是对 `wait_vllm_server.py` 和 `log_vllm_requests.py` 都改成 no-proxy opener
5. 等待脚本如果只会“傻等”，会放大排查成本：
   - 早期版本即使服务进程已经退出，也会一直等到超时
   - 后来给 `wait_vllm_server.py` 增加了 PID 存活检查，服务启动失败时可以更早暴露
6. 单看 `nvidia-smi` 不够，必须把运行窗口内的显存轨迹落盘：
   - 否则只能知道“某一刻大概占了多少”
   - 不能把请求窗口与显存峰值准确对应起来
   - 这也是后来补 `monitor_gpu_memory.py` 和 GPU summary 输出的原因

### 阶段四：双卡参数摸底

在阶段三确认双卡已经能稳定跑通后，没有直接开全量正式实验，而是先做了一轮双卡参数摸底，重点回答：

2. 哪组参数更适合作为正式双卡实验入口。

本轮对比重点放在两个方向：

- 打开 `enforce_eager`

它会复用双卡主入口 [run_stage1_dual_gpu.sh](/root/workspace/scripts/run_stage1_dual_gpu.sh)，便于后续重复运行同类摸底实验。

本轮摸底先在 `medium` 下比较了两组新参数：

- `offload4_graph`
- `offload8_eager`

随后把表现更好的组合继续拿到 `high` 下验证。

关键结果如下：

- 原始双卡基线 `offload8_graph`
  - `medium throughput = 6.0784`
  - `high throughput = 8.1765`
- `offload8_eager`
  - `medium throughput = 6.0956`
  - 启动更快，但请求期性能几乎没有改善
- `offload4_graph`
  - `medium throughput = 11.6342`
  - `high throughput = 15.8497`
  - `avg_ttft` 和 `avg_tpot` 也同步大幅下降

这里的结论非常关键：

1. `enforce_eager` 的主要收益更偏启动时间，而不是请求期性能。
2. 真正影响当前双卡性能的主要因素，是 `cpu_offload_gb=8` 带来的请求期搬运开销。
3. 当 `cpu_offload_gb` 从 `8` 降到 `4` 后，双卡高负载结果显著优于旧双卡基线，也显著优于单卡高负载结果。

这说明前面“双卡不如单卡”的现象并不是双卡本身无效，而是参数没有调到合适区间。

当前最值得继续作为正式双卡实验入口的配置是：

- `CUDA_VISIBLE_DEVICES=0,1`
- `tensor_parallel_size=2`
- `gpu_memory_utilization=0.70`
- `cpu_offload_gb=4`
- `max_model_len=1024`
- `disable_custom_all_reduce=True`
- `enforce_eager=False`
