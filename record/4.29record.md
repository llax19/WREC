## MoE-Lightning 模型与实验环境查证

操作：

1. 使用 `rg` 在工作区检索 `MoE-Lightning`，定位到本地论文抽取文件 `results/wrec/moe_lightning_extracted_20260427.txt`、相关分析记录和 HRM evaluator 脚本。
2. 阅读论文抽取文件中摘要、Evaluation Setup、Implementation、Models、Hardware、Workloads 相关段落。
3. 对照本地记录，区分“MoE-Lightning 使用的性能模型”和“论文实验中评估的 MoE LLM 模型”两个概念，避免把 HRM 与被推理的 LLM 混为一谈。

原理：

1. MoE-Lightning 是面向内存受限 GPU 的 MoE batch inference 系统，不是一个新的语言模型。
2. 系统用 HRM（Hierarchical Roofline Model）作为性能模型，结合硬件、模型、workload 和 policy，搜索更高吞吐的调度/放置策略。
3. 实验环境应从论文的 Evaluation Setup 提取，包括实现栈、模型、硬件配置、workload 和 baseline。

结论：

1. MoE-Lightning 的性能建模核心是 HRM；主要系统优化包括 CGOPipe、paged weights 和 CPU/GPU/I/O overlap。
2. 论文评估的 MoE LLM 包括 Mixtral 8x7B、Mixtral 8x22B 和 DBRX（132B，16 experts）。
3. 实现基于 PyTorch、vLLM、SGLang，使用 Python/C++，并基于 Intel MKL 实现定制 CPU GQA kernel。
4. 硬件设置包括单卡 T4 16GB、单卡 L4 24GB，以及 2/4 张 T4；CPU 为 Intel Xeon 24-core/192GB 或 32-core/416GB。
5. workload 包括 MTBench，以及 HELM 中的 Synthetic Reasoning 和 Summarization。

## Hugging Face 下载超时问题排查

操作：

1. 检查用户贴出的 traceback，定位主要异常为 `httpx.ReadTimeout` / `httpcore.ReadTimeout`，发生在 `huggingface_hub` 的 `snapshot_download` 多线程下载阶段。
2. 检查磁盘空间：根分区约 7.3T，已用 6.6T，剩余约 275G，使用率 97%。
3. 检查是否还有正在运行的下载进程，未发现活跃的 `hf download` / `huggingface` 下载进程。
4. 检查 `hf` CLI 版本与 Python 环境：`huggingface_hub` 为 1.12.2，Python 为 3.13.12，`hf_xet` 已安装，`hf_transfer` 未安装。
5. 检查本地 Mixtral 下载目录和 HF cache，发现 `/root/workspace/models/Mixtral-8x7B-Instruct-v0.1` 已占用约 87G，其中 `.cache/huggingface/download` 约 52G，包含多个可续传的 `.incomplete` 文件。
6. 使用 dry-run 验证更窄的下载策略：只包含 `*.safetensors`、`*.json`、`tokenizer.*`、`README.md`，并排除 `consolidated*.pt`。

原理：

1. `ReadTimeout` 表示下载流在默认超时时间内没有读到数据，常见于大文件、代理链路或 Hugging Face 多线程分片下载。
2. Hugging Face local-dir 下载会保留 `.incomplete` 临时文件，后续相同目标目录、相同 repo 的下载可以断点续传。
3. Mixtral 仓库同时包含 safetensors 权重和 Mistral 原始 `consolidated*.pt` 权重；对 vLLM/Transformers 场景通常只需要 safetensors 与 tokenizer/config，下载整个 repo 会浪费大量空间和时间。
4. 降低 `--max-workers` 并增大 `HF_HUB_DOWNLOAD_TIMEOUT` / `HF_HUB_ETAG_TIMEOUT` 可以降低代理或网络抖动导致的失败概率。

结论：

1. 这次失败的直接原因是网络读超时，不是模型目录必然损坏。
2. 本地已有部分完整 safetensors 分片，以及大量 `.incomplete` 分片，可继续续传。
3. 建议后续使用窄范围、低并发、长超时的命令继续下载 Mixtral safetensors，避免继续下载 `consolidated*.pt`。
