## 2026-04-25 MoE-affinity 服务端调度 prototype

### 背景

阅读 MoE-Infinity 与 PRE-ATTENTION 后，决定先验证一个低侵入的
`affinity_to_running_batch` 调度链路：不先改模型 forward，也不先训练在线 predictor，
而是让 scheduler 从外部 JSONL 读取 `request_id -> expert_signature`，在服务端
waiting queue 中对 top-K 候选做动态重排。

这轮目标不是证明最终 MoE locality 收益，而是验证：

1. vLLM 可以启动 `--scheduling-policy moe_affinity`。
2. waiting queue 能保持 Length/LTR-like 静态基础顺序。
3. scheduler 在真正取 waiting 请求时，能根据当前 running batch 动态计算 affinity。
4. 客户端保持 `dispatch_strategy=fcfs`，实验仍属于 server-side scheduler。

### 实现

vLLM patch:

- `/root/miniconda3/envs/vllm_moe/lib/python3.10/site-packages/vllm/v1/core/sched/policy.py`
  - 新增 `MoEAffinityPolicy`
  - 从 `VLLM_MOE_AFFINITY_SIGNATURE_FILE` 读取 sparse expert signature
  - 可选从 `VLLM_MOE_AFFINITY_SCORE_FILE` 或 `VLLM_LTR_SCORE_FILE` 读取基础 score
  - 静态 `sort_key` 用于 heap 入队
  - 动态 `dynamic_sort_key(request, running)` 用于 top-K rerank
- `/root/miniconda3/envs/vllm_moe/lib/python3.10/site-packages/vllm/v1/core/sched/request_queue.py`
  - 新增 `SchedulingPolicy.MOE_AFFINITY`
  - 新增 `MoEAffinityRequestQueue`
- `/root/miniconda3/envs/vllm_moe/lib/python3.10/site-packages/vllm/v1/core/sched/scheduler.py`
  - 新增 `_select_waiting_request_for_scheduling`
  - 对 `moe_affinity` 从 waiting/skipped waiting 的前 `VLLM_MOE_AFFINITY_TOPK`
    个候选里按动态 key 选请求
  - 被选请求用 `remove_request` 出队，避免只能 pop heap 顶
- `/root/miniconda3/envs/vllm_moe/lib/python3.10/site-packages/vllm/config/scheduler.py`
  - CLI 允许 `--scheduling-policy moe_affinity`

实验脚本:

- `/root/workspace/scripts/build_moe_affinity_signature_file.py`
  - 从请求 manifest 生成 proxy expert signature
  - 当前是 plumbing/proxy，不是真实 router trace
  - 默认对每层只保留 top-8 experts，避免 proxy signature 过密
- `/root/workspace/scripts/run_tp2_moe_affinity_server_formal.sh`
  - TP=2 服务端 MoE-affinity 实验入口
- `/root/workspace/scripts/log_vllm_requests.py`
  - `strategy` 允许 `moe_affinity`

同步更新 patch overlay:

- `/root/workspace/patches/vllm_policy_scheduler_20260424/`

### 动态排序键

最终采用保守 key：

```text
(
  priority,
  base_score_bucket,
  estimated_total_tokens_bucket,
  -affinity_weight * affinity_to_running_batch,
  base_score,
  estimated_total_tokens,
  prompt_tokens,
  arrival_time,
  request_id
)
```

其中：

- `base_score` 优先来自外部 score 文件，缺失时退化为 `prompt_tokens + max_tokens`
- `estimated_total_tokens_bucket` 是保护项，避免 affinity 在长度差异较大时覆盖
  Length-only 的 TTFT 收益
- `affinity_to_running_batch` 当前为 request signature 与 running aggregate signature
  的 cosine similarity

router trace 或 PRE-ATTENTION 式预测 signature。否则 affinity 信号本身没有足够物理意义，
不能期待超过 Length-only。

## 2026-04-25 OLMoE public route trace 离线 replay

### 目的

在还没采 Qwen1.5-MoE-A2.7B 真实 router trace 前，先用公开 MoE route trace 验证：

1. 真实 expert route signature 是否存在可被调度利用的 locality。
2. 按 `affinity_to_running_batch` 做局部重排，能不能减少同 batch 的 expert working set。
3. 这种机制是否值得继续接到我们论文里的 scheduler 设计。

这轮验证不替代 vLLM 端到端 latency 实验；它只回答机制可行性。

### 数据来源

- Hugging Face dataset: `allenai/analysis_olmoe`
- 数据集页面显示：
  - `train` split 共 `1,860` rows
  - 字段包含 `input_ids`, `predicted_token_ids`, `exp_ids`
  - 总文件大小约 `1.99 GB`
- 本轮没有下载完整 2GB 数据，而是通过 Hugging Face datasets-server rows API 抽样。
- OLMoE 配置按公开 Transformers 文档理解为：
  - `num_hidden_layers=16`
  - `num_experts=64`
  - `num_experts_per_tok=8`

### 新增脚本

- `/root/workspace/scripts/build_olmoe_trace_signatures.py`
  - 从 `allenai/analysis_olmoe` 的 rows API 读取 `exp_ids`
  - 自动识别本数据中的 route layout 为 `rank_layer`
  - 将 token-level expert ids 压成 request-level sparse signature
  - 本轮参数：
    - 每行均匀采样 `512` 个 route tokens
    - rank 权重为 `1 / (rank + 1)`
    - 每层保留 top-16 experts
    - 输出 `request_id -> experts`
- `/root/workspace/scripts/simulate_moe_affinity_trace_replay.py`
  - 纯 CPU replay
  - 对比 `fcfs`, `length`, `random`, `affinity`
  - 默认 batch size 用 `6`，对齐 high 档 `max_concurrency=6`
  - affinity 在 length base order 的 top-K 候选中按当前 batch aggregate signature 选 cosine 最高者
  - 指标：
    - `avg_batch_pairwise_cosine`: 同 batch 内 request signature pairwise cosine，越高越好
    - `avg_unique_experts_per_layer_per_batch`: 每层每 batch 触达的 unique experts，越低越好
    - LRU expert cache `misses_per_batch` / `hit_rate`: proxy expert weight cache locality

### n=192, batch=6, topK=32 结果

| policy | batch cosine | adjacent cosine | unique experts/layer/batch | cache hit@32 | miss/batch@32 | cache hit@48 | miss/batch@48 | avg displacement |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| FCFS | 0.460902 | 0.520440 | 38.293 | 0.057278 | 577.594 | 0.808375 | 117.406 | 0.000 |
| Length | 0.460902 | 0.520440 | 38.293 | 0.057278 | 577.594 | 0.808375 | 117.406 | 0.000 |
| Random | 0.425857 | 0.423432 | 40.348 | 0.035676 | 622.531 | 0.799109 | 129.688 | 55.375 |
| Affinity | 0.590893 | 0.620311 | 34.398 | 0.197593 | 441.625 | 0.829548 | 93.812 | 19.635 |

Affinity vs FCFS:

- batch 内 route cosine: `+28.20%`
- unique experts/layer/batch: `-10.17%`
- miss/batch@32: `-23.54%`
- miss/batch@48: `-20.10%`

解释：

- `length` 和 `fcfs` 完全相同，是因为这批 OLMoE rows 的 `input_tokens` 和
  `predicted_tokens` 都是 `2048`，`estimated_total_tokens=4096`，长度排序没有区分度。
- `random` 明显变差，说明原始顺序已经有一定 locality；affinity 仍能在此基础上继续改善。
- cache@64 的 miss/batch 对所有策略接近，因为每层一共只有 64 experts，容量接近全缓存时
  调度顺序不再是瓶颈。

### offset=512, n=96 sanity

为了排除“只在数据集开头有效”，又抽了 `offset=512` 的 96 条：

| --- | ---: | ---: | ---: | ---: |
| FCFS | 0.498675 | 37.719 | 560.188 | 111.438 |
| Length | 0.498675 | 37.719 | 560.188 | 111.438 |
| Random | 0.481016 | 38.574 | 584.875 | 110.750 |
| Affinity | 0.608421 | 34.066 | 410.938 | 100.250 |

Affinity vs FCFS:

- batch cosine: `+22.01%`
- unique experts/layer/batch: `-9.68%`
- miss/batch@32: `-26.64%`
- miss/batch@48: `-10.04%`

趋势与 n=192 主结果一致。

### topK sensitivity, n=192

| affinity topK | batch cosine | unique experts/layer/batch | miss/batch@32 | miss/batch@48 | avg displacement |
| ---: | ---: | ---: | ---: | ---: | ---: |
| 8 | 0.521407 | 36.258 | 514.875 | 118.750 | 5.292 |
| 16 | 0.565017 | 35.121 | 472.688 | 109.875 | 9.562 |
| 32 | 0.590893 | 34.398 | 441.625 | 93.812 | 19.635 |
| 64 | 0.614844 | 33.678 | 410.281 | 102.594 | 34.104 |

判断：

- `topK=8` 重排很保守，batch cosine 和 unique experts 有改善，但 cache@48 没有稳定收益。
- `topK=16/32` 是比较合理的区域，收益明显，重排幅度还可控。
- `topK=64` 进一步提高 locality，但 displacement 明显变大；这可能会伤害真实服务里的公平性、
  TTFT 和 length-based 优势。

### 当前结论

这轮离线 replay 支持我们的核心假设：

> 真实 route signature 中存在可调度的 expert locality；在长度基础顺序上做
> `affinity_to_running_batch` 局部重排，可以显著提高同 batch route 相似度，并降低
> expert working set/cache miss proxy。

因此，`affinity_to_running_batch` 值得继续放入论文 schedule 机制。但论文中需要明确区分：

1. Public OLMoE trace replay: 机制验证，证明 route locality signal 有效。
2. Qwen1.5-MoE-A2.7B real router trace + vLLM: 端到端 latency 验证，证明我们的系统收益。
3. PRE-ATTENTION predictor: 后续可作为“不需要真实 router trace oracle”的 signature 预测器。

### 限制

- OLMoE 不是 Qwen1.5-MoE-A2.7B，专家数、路由器、模型层数均不同。
- 这轮用的是 oracle route signature，不是在线可得 predictor。
- `analysis_olmoe` 每行长度基本一致，所以无法验证 length-vs-affinity 的真实 tradeoff。
- 指标是 expert locality proxy，不是 TTFT/TPOT/wall time。

### 下一步

1. 在我们的 Qwen/vLLM 环境里采真实 router trace，并复用同一个 replay simulator。
2. 如果 Qwen trace replay 也出现类似 locality 改善，再跑 vLLM `moe_affinity` 服务端策略。
3. 服务端默认建议从 `VLLM_MOE_AFFINITY_TOPK=16` 或 `32` 开始，并保留 length bucket，避免
   affinity 过度重排伤害 TTFT。
