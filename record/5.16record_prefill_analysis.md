## 2026-05-16 对 slot60 prefill 中高 U 行的解释

- 操作：复查 `wrec_expert_residency.py` 中 `begin_forward()`、`maybe_prefetch_for_attention()` 与 `_record_forward_stats()` 的调用顺序，并对 `runtime_qwen_real_table_a_prefill_n16_alpha1_slot60_wrec_only_20260516T010941Z/wrec_residency_stats.jsonl` 的 `forward_index=49..72` 做逐行抽样。

  - `forward_index=49..72` 中 `U=60`，说明该 prefill chunk 在这些层上 top-k 去重后覆盖了全部 60 个 experts。

## 2026-05-16 Clarification on `async_prefetch_copy_ms` and `waste_transfer_bytes`
- 现象：在 full-serving timing 样本里，很多行 `M>0`，但 `async_prefetch_copy_ms=null`，同时 `waste_transfer_bytes=0`。
- 解释：这两者并不矛盾。`M` 统计的是“当前 forward 进入 MoE 时仍缺失的 expert”，而 `waste_transfer_bytes` 统计的是“上一轮已经预取到 GPU、但本轮 current_experts 没用到的 expert”。如果根本没有发生 async prefetch，那么会同时出现 `M>0`、`async_prefetch_copy_ms=null`、`waste_transfer_bytes=0`。
- 原因：当前这 48 行主要覆盖第一个请求的 prefill 与第一个 decode token，sidecar 还没有形成可用的 ranking 候选，因此 `prefetched=[]`，`last_prefetched_experts` 为空，`_onload_experts_async()` 没有真正执行 copy，也就不会记录 async copy timing event。
- 结论：这更像统计口径容易误读，而不是直接的数值错误；但现有字段不足以区分“没尝试预取”和“尝试了但没有命中”。后续若继续做 timing，建议补 `prefetch_attempted` / `predicted_expert_count` / `prefetch_skip_reason` 这类字段。

## 2026-05-16 Correction on timing metric interpretation
- 更正：之前把 `prefetch_window_ms` 与 `async_prefetch_copy_ms` 直接比较，进而说“decode 的 copy 时间能被窗口容纳”，这个判断过头了。
- 原因：当前 instrumentation 里，`prefetch_window_ms = attention_start_event -> moe_ready_event`，其中 `moe_ready_event` 是在 `begin_forward()` 里 `_wait_for_prefetch(state)` 之后记录的，所以它天然混入了 MoE 入口等待时间；同时 `attention_start_event` 记录在 `maybe_prefetch_for_attention()` 之前，这段逻辑里还会走 `_claim_shared_slot_pool()`，可能把上一层/上一轮 prefetch 的等待也算进来。
- 结论：当前 `prefetch_window_ms` 不能被解释为“纯 attention 计算窗口”，`attention_ms` 也可能被 pool reclaim / wait_event 污染。因此现有数据只能说明 timing 事件的相对量级，不能直接证明 async copy 已经被 attention 完整覆盖。
- 后续修正方向：如果要严格回答 overlap 是否成立，需要补更细的事件边界，例如 `attention_body_start` / `attention_body_end`、`moe_wait_start` / `moe_wait_end`，把 attention 真正的计算时间和 MoE 入口等待时间拆开。

## 2026-05-16 Ranking-only scoring vs finite-slot assumption
- 结论：`SIDECAR_TOTAL_SLOTS` 并没有直接进入 ranking-only 的 score 公式。`rank_layer_experts()` 只是对每个 expert 调 `policy.score()`，而 `policy.score()` 只依赖 `base/recent/request/cross-layer`，以及 `cache.resident` 上的 contention；但 ranking-only 初始化的是空 cache，并且要求 `contention_penalty == 0`，所以 `total_slots` 不会实际影响排序分数。
- 真正的错位：sidecar 仍然输出“整层 full ranking”，runtime 再按 `active_slots` 截断使用；也就是说，不是 score 公式还停留在旧 finite-slot，而是 ranking 接口本身不是 budget-aware。要和现在的 prefetch 逻辑对齐，应该改 ranking contract，而不是只改 score 公式。


## 2026-05-16 Make ranking-only output budget-aware
- 操作：把 ranking-only 路径从“sidecar 生成 full ranking，runtime 再按 `active_slots` 截断”改成“runtime 先算出本次每层 budget，sidecar 只返回该 budget 下会被消费的 top-N ranking”。
- 改动：

  - 在 [runtime_engine.py](/root/WREC/scripts/wrec/runtime/runtime_engine.py) 解析 `ranking_budget_by_layer`，并在 `ranked_experts_payload()` 内按层裁剪 `ranked_experts_by_layer`，让 sidecar 实际输出和 runtime 消费预算一致。
- 原理：当前 ranking-only 模式下，真正的错位不是 `SIDECAR_TOTAL_SLOTS` 直接改坏了 score 公式，而是 ranking 接口仍然输出整层全排序，runtime 再做二次截断。这样 sidecar 不知道本次 prefetch 真正的消费上限，warm-start 还会留下与当前 budget 无关的 full ranking。把 budget 显式带进事件载荷后，ranking 合同才和当前 finite-slot/runtime-prefetch 语义对齐。
- 困难：一开始容易把问题归因到 `SIDECAR_TOTAL_SLOTS=192` 这类旧全局 budget 上，但本地核对 `policy.score()` 后确认 ranking-only 下分数并不直接依赖它；真正需要修的是接口预算而不是 score 公式。另一个问题是运行环境与源码目录分离，必须在本地源码改完后重新部署进 `vllm_moe` site-packages 才能确保后续实验不跑旧逻辑。

- 结论：现在 sidecar ranking 输出已经从“整层 full ranking”改为“按 runtime 当前 `active_slots` 预算裁剪后的 ranking”。这版更接近当前 ranking-only + finite-slot prefetch 的真实消费语义，后续实验可以直接观察这种 budget-aware ranking 会不会改善 decode 的 hit/waste 结构。

## 2026-05-16 Clarify timing-field semantics for `async_prefetch_copy_ms`
- 操作：追踪 `wrec_expert_residency.py` 与 attention hook，核对 `async_prefetch_copy_ms`、`prefetch_issue_to_attention_end_ms`、`attention_to_moe_wait_ms`、`moe_wait_ms` 的事件边界，并抽查 `runtime_qwen_real_table_b_full_n64_alpha1_wrec_only_timing_20260516T051932Z/wrec_residency_stats.jsonl` 末尾样本。
- 原理：`async_prefetch_copy_ms` 是 prefetch stream 上 `prefetch_copy_start_event -> prefetch_copy_end_event` 的完整异步 copy 时长；`moe_wait_ms` 才是主 stream 在 `begin_forward()` 里 `_wait_for_prefetch()` 前后记录到的显式等待时间；`attention_to_moe_wait_ms` 只是 attention 结束到开始 wait 之间的 gap，不是 wait 本身；`prefetch_issue_to_attention_end_ms` 则是 issue prefetch 之后到 attention 结束的重叠窗口。
- 样本：末行 `forward_index=792` 显示 `async_prefetch_copy_ms=66.08ms`、`prefetch_issue_to_attention_end_ms=0.35ms`、`attention_to_moe_wait_ms=0.44ms`、`moe_wait_ms=0.04ms`、`sync_onload_ms=32.71ms`，同时 `H=8`、`M=12`、`prefetched` 数量为 `24`。按 `sync_copy_bytes/M` 反推，每个 expert 约 `17.3MB`，这一行异步预取的数据量大约在 `24 * 17.3MB ≈ 415MB` 量级。
- 结论：不能把 `async_prefetch_copy_ms` 直接解释成“attention 结束后的等待时间”。如果要看 prefetch 导致的 attention 后显式阻塞，更接近 `moe_wait_ms`；而这行真正大的 post-attention 开销主要是 miss 后的 `sync_onload_ms`，不是 `moe_wait_ms`。
