## Qwen finite-slot runtime smoke

### 操作

- 使用 `train_n256.jsonl` 在 `18765` 启动 WREC sidecar：
  - `total_slots=192`
  - sidecar load/prior 时间约 `5.93s / 5.36s`
- 构造短 eval 子集：
  - `data/prompts/qwen_finite_slot_eval_short_n8_20260511.jsonl`
  - 从 eval split 中筛选 `approx_input_tokens <= 120` 的 8 条请求
  - 将 `target_max_new_tokens` 设为 `1`
- 启动并测试 Qwen finite-slot：
  - `slot_capacity=16`
  - `max_model_len=256`
  - `max_num_seqs=1`
  - `max_num_batched_tokens=32`
  - `WREC_SIDECAR_URL=http://127.0.0.1:18765`
  - `--offload-backend uva`
  - `--cpu-offload-gb 28`
  - `--cpu-offload-params experts.w13_weight experts.w2_weight`
- 失败后继续测试：
  - `slot_capacity=40, max_num_batched_tokens=32`
  - `slot_capacity=16, max_num_batched_tokens=4`

### 原理

- Qwen top-k 为 4。一次 FusedMoE forward 中当前 chunk 的 unique experts 必须全部能映射到 GPU slot。
- 当 chunk 过大时，即使每层只处理一个请求，也可能在一个 MoE 层中触达 40 个以上 unique experts。
- 因此 finite-slot 正确性不仅由 slot 数决定，也由 `max_num_batched_tokens` 决定。

### 困难与处理

- `slot=16, max_num_batched_tokens=32` 失败：
  - 第一条短请求完成
  - 第二条触发 `WREC expert residency slot overflow: 40 experts needed but capacity is 16 for layer model.layers.0.mlp.experts`
- `slot=40, max_num_batched_tokens=32` 仍失败：
  - 第二条触发 `slot overflow: 42 experts needed but capacity is 40 for layer model.layers.4.mlp.experts`
- 处理方式：
  - 将 `max_num_batched_tokens` 降到 `4`
  - 使用 `slot_capacity=16`
  - 该配置下 8 条短 eval 请求全部完成

### CPU backing 复用情况

- 即使显式设置：
  - `--offload-backend uva`
  - `--cpu-offload-gb 28`
  - `--cpu-offload-params experts.w13_weight experts.w2_weight`
- vLLM 日志仍显示：
  - `Total CPU offloaded parameters: 23.2`
  - 但 WREC 仍打印 `falling back to an extra CPU copy`
- 结论：
  - vLLM 确实执行了 UVA offload；
  - 但 WREC 触达的 `w13_weight/w2_weight` 对象没有成功复用 `_vllm_cpu_offload_storage`，或属性 shape/dtype/device 不满足当前复用条件；
  - 下一步需要在 `_cpu_backing_for_weight()` 中增加参数名、属性存在性、storage shape/device/dtype 的诊断日志，判断是 offloader 匹配问题、参数对象替换问题，还是 WREC 初始化时序问题。

### 结论

- Qwen 单卡真实 finite-slot 在 `slot=16, max_num_batched_tokens=4` 下可以完成短 eval 请求。
- `max_num_batched_tokens=32` 不适合当前 Qwen finite-slot smoke，因为单个 forward chunk 的 unique experts 超过 slot capacity。
- 当前结果是 controlled smoke，不是最终吞吐结论。
- 下一步优先级：
  1. 修复或解释 WREC CPU backing 不能复用 vLLM offload storage 的原因。
  2. 修复 finite-slot 路径下 routed expert export 计数不完整的问题。
  3. 在上述问题解决后再扩大 eval 规模和 slot/chunk sweep。

## finite-slot routed expert export 计数修复

### 操作

- 检查 finite-slot 使用的 live vLLM sidecar client：
  - `external/vllm-0.19.0/vllm/v1/core/sched/wrec_sidecar_client.py`
  - `~/miniconda3/envs/vllm_moe/lib/python3.10/site-packages/vllm/v1/core/sched/wrec_sidecar_client.py`
- 在 sidecar client 中补两类保护：
  - 当 `WREC_SIDECAR_MAX_EVENTS_PER_REQUEST` 截断导出时打印显式 warning，不再静默返回。
  - 单条 event 提交失败时先重建 HTTP 连接并重试一次，避免第一条成功后因 `RemoteDisconnected` 等连接错误直接 fail-open 停用 bridge。
- 对 live site-packages 版本执行 `py_compile`。
- 用本地 HTTP smoke 人工制造“首条请求断连”：
  - 第一条 POST 故意断开连接；
  - 验证 patched client 会重连并继续送完剩余 routed events。

### 原理

- 先前 finite-slot 只统计到 `router_events=1, expert_refs=4`，与“首条 event 成功、后续提交因连接错误停止”或“单 request 被 `max_events_per_request=1` 截断”两种模式完全一致。
- 原 sidecar client 对这两种情况都不够健壮：
  - 截断是静默的；
  - 提交异常会直接禁用 bridge，后续 event 全部丢失。
- 修复后，即使 sidecar 短暂断连，client 也会在当前 request 内自动重连一次并继续提交；若配置了导出上限，也会在日志中直接暴露截断事实。

### 困难与处理

- 结果目录里没有保留 finite-slot serving 原始日志，无法直接从当天日志确认是“截断”还是“断连”。
- 处理方式：
  - 先从现象反推最可能的两个静默漏计分支；
  - 再对 sidecar client 同时加上“显式告警 + 重连重试”；
  - 最后用本地断连 smoke 验证该修复确实能覆盖“第一条成功、后续连接断开”的失败模式。

### 结论

- finite-slot routed expert export 的静默漏计风险已在 sidecar client 层修复。
- 本地 retry smoke 已验证：
  - 首次 `RemoteDisconnected` 后会自动重连；
  - `shape=[2, 2, 2]` 的 routed experts 最终完整送达 `4` 条 event；
  - client 不会在第一次断连后被永久禁用。
- 尚未重新跑 Qwen finite-slot n=8 短测，因此本轮还不能直接把 `router_events=1, expert_refs=4` 改判为已实测恢复；需要下轮复跑确认。

## finite-slot sidecar fix 后复跑观察

### 操作

- 重新启动 Qwen finite-slot guarded smoke：
  - sidecar `port=18765`
  - `total_slots=192`
  - `slot_capacity=16`
  - `max_num_batched_tokens=4`
  - `WREC_SIDECAR_MAX_EVENTS_PER_REQUEST=0`
- 按 `data/prompts/qwen_finite_slot_eval_short_n8_20260511.jsonl` 顺序重放 8 条短 eval 请求。
- 输出：
  - `results/wrec/runtime_finite_slot_qwen_single_gpu_20260511/qwen_finite_slot_short_eval_n8_rerun_after_sidecar_fix_20260511.json`
  - `results/wrec/runtime_finite_slot_qwen_single_gpu_20260511/qwen_finite_slot_short_eval_n8_rerun_after_sidecar_fix_20260511.md`

### 原理

- 这次复跑的目标不是重新比较 latency，而是验证 sidecar client 修复后 finite-slot routed expert export 是否恢复完整计数。
- 因此显式设置 `WREC_SIDECAR_MAX_EVENTS_PER_REQUEST=0`，排除 request-level event limit 截断因素。

### 结果

- 8 条请求全部完成：
  - mean latency `2.235494 s`
  - max latency `3.565797 s`
  - input tokens `117`
  - output tokens `8`
- sidecar metrics 仍停在第一条 event：
  - `router_events=1`
  - `expert_refs=4`
  - `shadow_hits=1`
  - `shadow_misses=3`
  - `would_admit=3`
  - `would_evict=3`

### 困难与处理

- 这次不是 sidecar HTTP 断连或静默截断，而是第一条 sidecar response 回到 finite-slot manager 时触发异常：
  - `RuntimeError: Inplace update to inference tensor outside InferenceMode is not allowed`
- live 栈显示：
  - `record_sidecar_response()` 处理 `would_evict`
  - 调用 `_evict_expert()`
  - 在 `state.expert_map[expert_id] = -1` 处报错
- 由于 `submit_routed_experts()` 外层仍是 fail-open，该异常会直接触发 `WREC sidecar bridge failed`，随后 bridge 被禁用，因此后续 routed expert events 不再上报。

### 结论

- sidecar client reconnect patch 本身没有被这次复跑证伪；但它也不是当前 finite-slot 计数停在 `1/4` 的主阻塞。
- 当前真正阻塞是 finite-slot eviction path 在 sidecar response handling 中对 `expert_map` 做 inference tensor 原地写。
- 在修复 `wrec_expert_residency.py` 这条 eviction/update 路径前，不能把 finite-slot sidecar metrics 当成完整 trace 统计。

## finite-slot expert_map InferenceMode 修复

### 操作

- 修改 finite-slot residency manager 两份代码：
  - `external/vllm-0.19.0/vllm/model_executor/layers/fused_moe/wrec_expert_residency.py`
  - `~/miniconda3/envs/vllm_moe/lib/python3.10/site-packages/vllm/model_executor/layers/fused_moe/wrec_expert_residency.py`
- 新增：
  - `_new_expert_map()`：在 `torch.inference_mode(False)` 下创建 `expert_map`
  - `_set_expert_map_value()`：统一在 `torch.inference_mode(False) + torch.no_grad()` 下更新 slot 映射
- 将 `state.expert_map[expert_id] = slot_id/-1` 两处原地写都改为 helper。
- 重新复跑 Qwen finite-slot 短测：
  - `slot_capacity=16`
  - `max_num_batched_tokens=4`
  - `WREC_SIDECAR_MAX_EVENTS_PER_REQUEST=0`
- 输出：
  - `results/wrec/runtime_finite_slot_qwen_single_gpu_20260511/qwen_finite_slot_short_eval_n8_rerun_after_expert_map_fix_20260511.json`
  - `results/wrec/runtime_finite_slot_qwen_single_gpu_20260511/qwen_finite_slot_short_eval_n8_rerun_after_expert_map_fix_20260511.md`

### 原理

- 上轮阻塞点不是 sidecar HTTP export，而是 finite-slot manager 在处理 `would_evict` 时对 `expert_map` 做原地写，撞上 PyTorch `InferenceMode` 限制。
- 该问题的关键不是“是否允许单次写入”，而是 `expert_map` 的生命周期本身属于 finite-slot 运行时可变状态，因此不应作为 inference tensor 创建或更新。
- 把创建和更新都移到 `InferenceMode` 之外，才符合它作为逻辑 expert 到 GPU slot 映射表的语义。

### 困难与处理

- 仅修 eviction 点还不够，因为 onload 路径也会写 `expert_map`。
- 处理方式：
  - 不做局部热补丁；
  - 改成统一 helper；
  - 确保创建和所有写点使用同一套可变语义。

### 结果

- 修复后 8 条请求全部完成：
  - mean latency `3.669668 s`
  - max latency `5.298327 s`
  - input tokens `117`
  - output tokens `8`
- sidecar metrics 恢复完整：
  - `router_events=2808`
  - `expert_refs=11232`
  - `shadow_hits=2670`
  - `shadow_misses=8562`
  - `would_admit=8043`
  - `would_bypass=519`
  - `would_evict=8043`
- 本轮没有再复现 `WREC sidecar bridge failed` 或 `InferenceMode` 原地写异常。

### 结论

- 对当前 guarded Qwen finite-slot 配置而言，routed expert export 计数不完整问题已经修复。
- 这意味着 `slot=16, max_num_batched_tokens=4` 这条路径现在可以作为更大规模 finite-slot 实验的起点。
- 但 CPU backing 仍未复用 vLLM offload storage；扩大实验时仍需关注 host memory 和额外 CPU copy 成本。
