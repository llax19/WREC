\section{相关工作与技术背景}

本章介绍本文后续问题建模、方法设计和实验分析所依赖的背景知识与相关研究。按照本文的研究目标，本章重点围绕混合专家模型的稀疏路由机制、大语言模型推理系统中的显存管理、资源受限场景下的权重卸载，以及 MoE 专家缓存与预取优化展开。相关内容均服务于后文的专家访问序列建模和 WREC 专家缓存策略设计，不展开本文个人工作的具体算法细节。

\subsection{混合专家模型与稀疏路由机制}

混合专家模型（Mixture-of-Experts, MoE）的核心思想是在神经网络中引入多个相对独立的专家子网络，并通过路由器为不同输入动态选择其中一部分专家参与计算。早期稀疏门控 MoE 工作表明，在只激活少量专家的条件下，可以显著扩大模型参数规模，从而提高模型容量而不线性增加每个 token 的计算量~\cite{shazeer2017outrageously}。Switch Transformer 进一步将每个 token 路由到单个专家，以简化稀疏训练与分布式通信开销，说明 MoE 结构可以作为大规模语言模型扩展的重要路线~\cite{fedus2022switch}。

在典型 Transformer MoE 层中，原本的前馈网络（Feed-Forward Network, FFN）被替换为多个专家 FFN。给定输入 token 表示 $x$，路由器首先计算该 token 对不同专家的选择分数，然后选择 top-$k$ 个专家参与计算。一个简化的 MoE 层输出可以表示为：

\begin{equation}
    \label{equ:moe-layer}
    y(x)=\sum_{i\in \mathcal{S}(x)} g_i(x)E_i(x),
\end{equation}

其中，$\mathcal{S}(x)$ 表示路由器为输入 $x$ 选出的专家集合，$E_i(\cdot)$ 表示第 $i$ 个专家网络，$g_i(x)$ 是对应的路由权重。该表达式说明，MoE 的计算路径由路由器动态决定；即使模型参数总量很大，单个 token 在一次 MoE 层计算中也只访问其中少数专家。

以 Mixtral 为代表的开源 MoE 大模型采用 top-$2$ 路由，在每个 MoE 层为每个 token 选择两个专家参与计算~\cite{jiang2024mixtral}。这种设计带来了两个与系统部署直接相关的特征。第一，专家权重占据模型参数的主要部分，全部专家常驻 GPU 会对显存容量提出较高要求。第二，路由选择会把推理过程转化为 token、layer 和 expert 共同决定的动态专家访问序列。如果系统不能使所有专家常驻 GPU，那么每一次专家访问都可能对应一次缓存查询，缓存未命中则会触发专家权重从 CPU 内存到 GPU 显存的传输。

MoE 路由机制还影响专家访问的局部性。对于一个请求内部的相邻 token，路由器可能反复选择相同或相近专家；对于相邻层，专家选择也可能存在统计上的转移关系；对于一个 workload，不同请求之间又可能形成相对稳定的专家热度分布。本文后续的 WREC 策略正是围绕这些可被在线利用的访问结构展开，但本章只介绍其背景动机，具体统计定义和策略打分方式将在后续章节给出。

\subsection{大语言模型推理系统中的显存管理}

大语言模型推理通常包含 prefill 和 decode 两个阶段。prefill 阶段一次性处理输入提示词，生成初始 KV cache；decode 阶段逐 token 生成输出，并不断追加新的 KV cache。二者对系统资源的压力不同：prefill 阶段通常具有较大的并行计算量和较密集的 token 访问，decode 阶段则更容易受到自回归串行生成和请求调度影响。因此，推理系统需要同时管理模型权重、KV cache、临时激活和批处理调度。

连续批处理是提升大模型 serving 吞吐的重要技术。Orca 将请求级调度推进到迭代级调度，使不同请求可以在生成过程中动态进入和离开批次，从而提高 GPU 利用率~\cite{yu2022orca}。在此基础上，后续系统进一步关注服务质量目标、长度预测和请求排序等调度问题。例如，学习排序式调度和基于过去-未来信息的调度方法尝试在吞吐、延迟和 SLA 之间取得平衡~\cite{fu2024efficient,gong2024pastfuture}。这些工作主要改变请求排队和 batch 组织方式，而本文关注的对象更低一层：给定路由结果后，专家权重是否已经驻留在 GPU 上，以及缓存未命中造成的传输停顿如何降低。

显存管理是推理系统的另一条关键主线。PagedAttention 将 KV cache 组织为分页块，减少 KV cache 连续分配带来的外部碎片，并支持更灵活的请求调度和内存复用~\cite{kwon2023pagedattention}。DeepSpeed-Inference 则从张量并行、算子融合、通信优化等角度提高大模型推理效率~\cite{aminabadi2022deepspeed}。这些系统说明，大模型推理优化并不只依赖单个算子速度，还依赖参数、KV cache 和执行计划在硬件资源之间的协调。

对于 MoE 模型，显存管理问题进一步复杂化。稠密模型的权重访问在每一层基本固定，而 MoE 模型的专家权重访问由路由器动态决定。当专家总权重超过可用 GPU 显存时，系统需要选择一部分专家常驻 GPU，其余专家在访问时按需加载。此时，专家缓存管理与 KV cache 管理共同竞争有限显存，但二者的访问模式不同：KV cache 与请求上下文长度紧密相关，专家权重则与路由事件序列紧密相关。因此，直接套用通用 KV cache 管理策略并不能完整解决 MoE 专家驻留问题。

\subsection{资源受限推理中的权重卸载}

当模型参数或运行时状态超过 GPU 显存容量时，权重卸载（offloading）成为资源受限推理的重要手段。FlexGen 面向单 GPU 环境，联合 GPU、CPU 和磁盘空间，为模型权重、KV cache 和激活设计静态调度与卸载策略，以提高大模型离线推理吞吐~\cite{sheng2023flexgen}。PowerInfer 观察到部分神经元在本地推理中更频繁激活，并尝试将热权重保留在 GPU 上、冷权重放置在 CPU 上，从而改善消费级 GPU 上的推理效率~\cite{song2023powerinfer}。LLM in a Flash 则关注有限内存设备上的权重加载与稀疏访问问题，通过减少数据搬移支持更大模型在内存受限设备上运行~\cite{alizadeh2024llminflash}。

这些工作与本文共享一个基本前提：在资源受限环境下，数据搬移本身会成为推理性能的重要瓶颈。不同之处在于，通用稠密模型卸载通常面对的是层级或张量级的相对固定访问顺序，而 MoE 专家卸载面对的是路由器产生的动态专家访问序列。对于稠密 Transformer，若给定模型结构和执行阶段，下一层权重访问基本可以由计算图直接确定；对于 MoE，下一次访问哪个专家取决于当前 token 的路由结果，且不同请求、不同层之间可能存在复杂统计关系。

资源受限推理中的另一个关键问题是传输与计算能否重叠。如果权重传输可以完全隐藏在计算之后，则缓存未命中的性能影响较小；如果 CPU-GPU 带宽不足，或专家权重较大而无法及时预取，则未命中会直接转化为停顿。本文后续使用硬件资源模型判断目标配置是否处于 expert-transfer-bound 状态，其目的正是区分“专家传输是主要瓶颈”的场景与“专家传输可以被其他计算掩盖”的场景。只有在前者中，专家缓存调度才具有明确的优化价值。

\subsection{MoE 专家卸载、缓存与预取相关研究}

近年来，面向 MoE 模型的专家卸载和缓存优化逐渐成为推理系统研究的重要方向。Fast MoE offloading 相关工作围绕 Mixtral 等模型讨论了专家按需加载、缓存和异步预取问题，说明在单卡或有限显存环境下，专家权重不能简单视为始终常驻的静态参数~\cite{eliseev2023offloading}。MoE-Infinity 进一步从激活感知角度设计专家卸载策略，希望根据专家实际激活情况减少不必要的数据搬移~\cite{xue2024moeinfinity}。MoE-Lightning 则关注内存受限 GPU 上的高吞吐 MoE 推理，强调专家放置、传输和执行调度之间的联合优化~\cite{cao2025moelightning}。

上述研究说明，MoE 专家管理包含三个相互关联的问题。第一是 resident set 选择，即在有限 GPU 显存中保留哪些专家。第二是 eviction 策略，即当新专家需要加载而缓存已满时，应该换出哪个专家。第三是 prefetch 策略，即能否在专家真正被访问前提前加载，减少后续等待时间。这三个问题与经典缓存问题类似，但 MoE 场景具有更强的结构性：专家访问不仅有时间顺序，还包含层号、请求编号、token 位置和 top-$k$ 路由结果等上下文。

现有 MoE 专家卸载工作通常依赖某种形式的专家热度、路由预测或执行流水线。静态热度可以利用训练集或历史 workload 统计，但当请求内部局部性较强时，单纯热度排序可能无法充分利用短期复用。最近最少使用（Least Recently Used, LRU）等在线缓存策略可以利用近期访问，但它不区分同一请求内部的结构性复用、跨层转移关系和 workload 级先验。预取策略如果能够准确预测未来专家访问，则可以减少停顿；但如果预测错误或预取过多，也会带来额外传输和缓存污染。

也有研究从模型是否适合专家卸载的角度分析 MoE 路由一致性。相关工作指出，不同 MoE 模型的本地路由稳定性并不相同，因此并非所有 MoE 都同样适合专家卸载优化~\cite{liang2026notallmodels}。这一观察与本文的问题设定一致：在设计专家缓存策略前，需要先分析目标模型和 workload 的专家访问局部性，而不能默认任何 MoE 都存在同等可利用的缓存收益。

\subsection{缓存策略与专家访问局部性}

从抽象角度看，专家缓存可以被视为带有传输成本的在线缓存问题。系统维护一个容量有限的专家 resident set；当访问的专家已经驻留在 GPU 上时，访问命中；当专家不在 resident set 中时，系统需要加载该专家，并在容量不足时选择已有专家换出。若每个专家权重大小近似相同，则缓存容量可以用可容纳的专家数量表示；若专家大小或传输代价不同，则需要进一步考虑加权容量和异构传输成本。

经典 LRU 策略假设最近访问过的对象更可能再次访问，因此在缓存满时换出最久未访问对象。静态热度策略则假设长期高频对象更值得常驻。二者分别对应时间局部性和频率局部性，但 MoE 专家访问还包含请求内局部性与跨层结构。举例来说，同一请求的一段输入 token 可能反复激活少数专家；某一层的专家选择也可能对下一层专家选择提供统计线索。若缓存策略只利用全局频率或最近访问时间，就可能遗漏这些上下文信息。

本文后续采用 trace-driven replay 对不同策略进行比较，其基本思想是将真实或离线采集到的路由结果整理为专家引用序列，然后在统一缓存预算下重放该序列，统计命中、未命中、传输量和停顿代理指标。该方法有两个优点。第一，它能够在不改变模型输出的前提下隔离专家缓存策略本身的影响。第二，它允许在同一条访问序列上比较在线策略、静态策略和离线 oracle-style 上界，从而判断当前策略是否真正缩小了可利用空间。

需要强调的是，离线最优或使用未来信息的策略只能作为上界和压力测试，不能直接视为可部署方案。实际 serving 系统中，缓存策略只能使用当前及过去可见的信息，或者使用来自训练集、历史 workload 的统计先验。因此，本文在相关实验中区分公平在线基线、静态先验基线和 oracle-style 参考结果，并将 WREC 定位为不改变路由结果、仅使用可在线维护状态的专家缓存调度方法。

\subsection{本文工作的定位}

与直接优化模型结构、修改路由器或重新训练 MoE 不同，本文将模型路由结果视为给定输入，研究在不改变模型输出的前提下如何降低专家权重搬移开销。与只报告系统端到端吞吐的工作不同，本文首先使用 trace-driven simulator 隔离分析专家缓存策略本身的贡献，并通过硬件资源模型限定结论适用的 transfer-bound 场景。这样的定位使得后续章节能够围绕三个问题展开：目标模型是否存在专家传输瓶颈，路由 trace 是否包含可利用局部性，以及 WREC 在线策略能够相对公平基线缩小多少缓存调度空间。

\subsection{本章小结}

本章介绍了 MoE 稀疏路由、大模型推理显存管理、资源受限权重卸载、MoE 专家缓存与预取，以及专家访问局部性等背景知识。现有研究表明，MoE 模型能够通过稀疏激活扩大模型容量，但在显存受限环境中，专家权重访问会转化为动态缓存和传输问题；同时，通用推理系统和通用卸载系统不能完全覆盖 MoE 专家级访问的结构性。基于这些背景，本文后续将进一步定义资源受限 MoE 专家缓存问题，并围绕真实路由 trace 设计和评估 WREC 在线专家缓存策略。
