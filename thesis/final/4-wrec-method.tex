\section{WREC 专家缓存策略设计与实现}

第 3 章已经将资源受限 MoE 推理抽象为专家引用序列上的在线缓存问题。本章在此基础上介绍 WREC（Workload-aware Router-event Expert Cache）专家缓存策略的设计与实现。WREC 的基本思路是：首先从历史路由轨迹中提取可复用的工作负载先验，然后在在线事件流中维护当前请求和当前 token 的局部状态，最后在缓存未命中时根据统一评分函数决定专家准入、绕过与换出。该策略不修改模型参数、路由器输出或生成结果，而只作用于专家权重的驻留集合选择。

本章的组织如下。首先说明 WREC 的设计目标和信息边界；随后介绍路由事件采集、专家引用展开和工作负载先验构造方法；接着说明在线状态维护与 WREC-H2 评分函数；最后给出基线策略、重放模拟器以及运行时接口的实现边界。这样的组织方式对应本文的主线：先明确哪些信息可以被在线策略使用，再说明这些信息如何进入缓存决策，最后说明如何在可控重放和运行时原型中验证该决策过程。

\subsection{设计目标与信息边界}

WREC 面向专家传输受限、部分专家驻留的 MoE prefill 场景。在该场景中，GPU 显存无法稳定容纳全部专家对象，系统需要在有限专家缓存容量下选择一部分专家驻留。当访问专家已经驻留时，推理过程可以直接执行对应 FFN；当访问专家未驻留时，系统需要从 CPU 内存加载专家权重，并可能产生额外传输等待。因此，WREC 的直接目标是在固定专家缓存预算下减少需求加载次数和专家传输量。

为避免将离线上界误写成可部署策略，本文对 WREC 的可用信息作如下限制。第一，WREC 可以使用训练侧或历史工作负载轨迹统计得到的先验，例如专家热度、短窗口出现概率和跨层转移概率。第二，WREC 可以使用当前请求中已经发生的路由事件，以及当前 token 在较早层已经产生的专家选择。第三，WREC 不使用 eval 轨迹中的未来真实访问，也不使用当前 token 后续层尚未发生的路由结果。除单独标记为 Belady 或 route-window oracle 的基线外，本文正文中的 WREC 主策略均满足上述在线信息约束。

在决策形式上，WREC 采用准入--绕过机制。对于一次未命中访问，系统必须临时获得该专家权重以完成当前计算；但该专家是否进入长期驻留集合仍由策略决定。如果候选专家的评分高于当前缓存中最不值得保留的专家，WREC 将其准入并换出低分专家；否则，该专家只服务于当前访问，随后不进入驻留集合。该机制用于避免低价值的一次性专家访问污染有限缓存。

\subsection{路由轨迹采集与专家引用展开}

WREC 的输入来自 MoE 路由事件。对于支持输出 router logits 的模型，本文在 prefill 阶段对请求进行前向计算，读取每一层路由器输出，并根据模型配置选择 top-$k$ 个专家。一次 token 级路由事件表示为

\begin{equation}
    \label{equ:wrec-router-event-detail}
    r=(q,p,l,\mathcal{E}_{q,p,l},\mathcal{P}_{q,p,l}),
\end{equation}

其中，$q$ 表示请求编号，$p$ 表示 token 在请求中的位置，$l$ 表示 MoE 层编号，$\mathcal{E}_{q,p,l}$ 表示该 token 在该层被选中的专家集合，$\mathcal{P}_{q,p,l}$ 表示对应路由概率。对于采用 top-$2$ 路由的 Mixtral，一个路由事件会展开为两个专家引用。

为了与第 3 章的问题建模保持一致，本文将路由事件展开为按时间排序的专家引用序列。每个引用对象记为 $(l,e)$，其中 $l$ 为层编号，$e$ 为该层中的专家编号。采用 $(l,e)$ 而不是单独的专家编号 $e$ 是必要的，因为不同 MoE 层中的同名专家并不共享权重。例如，第 $3$ 层 expert $1$ 与第 $4$ 层 expert $1$ 是两个不同的缓存对象。

轨迹采集过程保留请求编号、阶段、层号、token 位置、被选专家、路由概率和事件编号等字段。阶段字段用于区分 prefill 与 decode；本文主实验只使用 prefill 轨迹。事件编号用于验证运行时事件流与离线重放序列之间的顺序一致性。采集后的 JSONL 轨迹被转换为内部 \texttt{ExpertRef} 表示，每条记录包含引用序号、事件序号、请求编号、层号、token 位置和专家编号。

\subsection{工作负载先验构造}

WREC 使用训练侧路由轨迹构造三类工作负载先验。第一类是短窗口出现概率。设第 $l$ 层按时间顺序排列的路由事件窗口大小为 $w$，专家 $e$ 在一个窗口中至少出现一次的概率定义为

\begin{equation}
    \label{equ:wrec-window-prob}
    P_{\mathrm{win}}(l,e)=
    \frac{\#\{\mathrm{window}:e\in \mathrm{window}\}}{\#\{\mathrm{window}\}}.
\end{equation}

该指标反映专家在短时间范围内被再次访问的可能性。若某专家在多个窗口中频繁出现，则保留该专家可能减少后续需求加载。

第二类是窗口内期望路由次数。设 $C_{\mathrm{win}}(l,e)$ 表示专家 $e$ 在第 $l$ 层一个窗口中被引用的次数，则

\begin{equation}
    \label{equ:wrec-window-count}
    E_{\mathrm{win}}(l,e)=\mathbb{E}[C_{\mathrm{win}}(l,e)].
\end{equation}

结合单次专家加载传输时间 $T_{\mathrm{load}}$，本文为每个专家对象构造基础收益分

\begin{equation}
    \label{equ:wrec-base-score}
    B(l,e)=P_{\mathrm{win}}(l,e)\cdot E_{\mathrm{win}}(l,e)\cdot T_{\mathrm{load}}-T_{\mathrm{load}}.
\end{equation}

该分数近似表示：若加载并保留一个专家，未来短窗口内可能节省的传输时间与当前加载成本之间的差值。本文主实验中同一模型内各专家大小相同，因此 $T_{\mathrm{load}}$ 在同一配置下为常数；保留该形式是为了使方法能够扩展到专家大小或传输代价不同的场景。

第三类是跨层转移概率。对于同一请求、同一 token，若第 $l-1$ 层选择了专家 $u$，第 $l$ 层选择了专家 $e$，则统计二者在训练轨迹中的共现次数，并定义

\begin{equation}
    \label{equ:wrec-cross-prob}
    P_{\mathrm{cross}}^{(l)}(e\mid u)=
    \frac{N^{(l)}(u,e)}{\sum_{e'}N^{(l)}(u,e')}.
\end{equation}

当某个前层专家在训练轨迹中没有足够样本时，实现中使用均匀分布作为退化处理，以避免未见组合导致评分异常。跨层转移概率的作用是利用当前 token 已经产生的上一层路由结果，估计下一层哪些专家更可能被访问。

\subsection{在线状态维护}

在线阶段，WREC 维护三类状态。第一类是每层最近访问历史。系统为每一层维护长度为 $h$ 的专家引用队列，并同步维护队列内专家计数。由此可得到最近访问概率

\begin{equation}
    \label{equ:wrec-recent-prob}
    R_t(l,e)=\frac{\#\{e\ \mathrm{in\ recent\ queue\ of\ layer}\ l\}}{h}.
\end{equation}

该状态用于消融实验和后续扩展。第 5 章的实验表明，在本文的 prefill total-budget 设置中，recent 项不是稳定的主要收益来源，因此主策略将其权重设为 $0$。

第二类是当前请求内统计。对于正在处理的请求，WREC 按层维护该请求已经访问过的专家计数和总引用数。请求内概率定义为

\begin{equation}
    \label{equ:wrec-request-prob}
    Q_t(l,e)=
    \frac{\#\{e\ \mathrm{visited\ in\ current\ request,\ layer}\ l\}}
    {\#\{\mathrm{expert\ refs\ in\ current\ request,\ layer}\ l\}}.
\end{equation}

该项刻画同一输入请求内部的局部复用。与全局静态热度不同，请求内统计会随当前输入内容变化；与 LRU 不同，它在评分时显式区分层号和专家编号。

第三类是 token 层级历史。WREC 在处理同一 token 的不同层时，记录已经完成层的专家选择。当第 $l$ 层的专家 $e$ 被评分时，若同一 token 在第 $l-1$ 层选择了专家集合 $\mathcal{E}_{l-1,p}$，则跨层分数为

\begin{equation}
    \label{equ:wrec-cross-score}
    X_t(l,e)=
    \frac{1}{|\mathcal{E}_{l-1,p}|}
    \sum_{u\in \mathcal{E}_{l-1,p}} P_{\mathrm{cross}}^{(l)}(e\mid u).
\end{equation}

该项只使用当前 token 已经产生的上一层路由结果，不使用未来层真实访问，因此仍满足在线策略的信息约束。

\subsection{WREC-H2 评分函数与缓存决策}

WREC 对候选专家对象 $(l,e)$ 的评分定义为

\begin{equation}
    \label{equ:wrec-score}
    \mathrm{Score}_t(l,e)=
    B(l,e)+
    \alpha R_t(l,e)+
    \beta Q_t(l,e)+
    \gamma X_t(l,e)-
    \delta D_t(l,e),
\end{equation}

其中，$B(l,e)$ 是训练轨迹构造的基础分，$R_t(l,e)$ 是最近访问项，$Q_t(l,e)$ 是请求内局部项，$X_t(l,e)$ 是跨层转移项，$D_t(l,e)$ 是可选的驻留竞争惩罚项。本文主实验采用的配置为

\begin{equation}
    \label{equ:wrec-main-config}
    w=4,\quad h=8,\quad
    \alpha=0,\quad
    \beta=1024,\quad
    \gamma=1024,\quad
    \delta=0.
\end{equation}

其中，$w$ 是训练侧窗口大小，$h$ 是 recent-history 长度。将 $\alpha$ 置为 $0$ 的原因来自第 5 章消融实验：recent-history 在当前 prefill total-budget 重放中并不稳定提供正收益，而请求内与跨层信号更可靠。

当访问 $(l,e)$ 且该专家已经驻留时，WREC 将其记为命中，并更新最近访问、请求内统计和 token 层级历史。当访问未命中时，系统首先将该次访问计为一次需求加载，因为真实执行必须获得该专家权重才能完成当前 MoE 层计算；随后 WREC 决定是否让该专家进入长期驻留集合。如果缓存未满，则新专家直接进入驻留集合；如果缓存已满，则 WREC 计算当前驻留专家的分数，选取分数最低者作为候选换出对象。若新专家分数高于该候选换出对象，则执行换出并准入新专家；否则，新专家在本次需求加载后被绕过，不进入驻留集合。

这一准入--绕过机制区别于普通 LRU。LRU 在每次未命中后都会把新对象放入缓存，并换出最久未访问对象；WREC 则允许低分专家只为当前访问临时加载，而不污染后续驻留集合。对于 MoE prefill 轨迹，这一点较为重要：某些专家可能由于单个 token 的路由偶然被访问，但未必值得占用全局有限的专家缓存槽位。

\subsection{基线策略与离线上界}

为了公平评估 WREC，本文实现了以下基线。第一，\texttt{on\_demand} 不保留专家缓存，每次专家访问都视为需求加载，用于给出最朴素的传输压力。第二，LRU 使用在线最近访问顺序，未命中时换出最久未访问专家。第三，\texttt{static\_hot} 只使用训练轨迹中的全局专家频率，将最热的专家对象初始化为驻留集合，并不根据 eval 未来信息调整。第四，\texttt{route\_window\_prefetch} 使用 eval 未来窗口进行预取，因此只作为使用未来信息的压力基线，不作为可部署策略。第五，Belady oracle 使用完整未来访问序列选择最晚再次使用或不再使用的专家作为换出对象，是等大小缓存对象假设下的离线上界。

在容量分配上，本文区分 per-layer 容量和 total-budget 容量。per-layer 容量为每个 MoE 层分配相同专家槽数，便于与传统分层缓存直觉比较；total-budget 容量只规定全模型专家槽总数，允许热点层和热点专家在全局范围内竞争槽位。WREC 主结果使用 total-budget 模式，因为该模式更接近显存预算的真实约束：GPU 显存只关心总驻留专家权重大小，而不要求每一层拥有完全相同的槽数。

\subsection{重放模拟器实现}

本文实现了基于访问轨迹的重放模拟器，用于隔离评估缓存策略本身。模拟器输入为 eval 路由轨迹、train 路由轨迹、模型专家大小、CPU-GPU 有效带宽和缓存预算。对每条专家引用，模拟器按策略更新缓存状态，并统计需求命中、需求未命中、需求加载、预取加载、预取浪费、传输字节数和传输时间估计值。

对无预取策略而言，需求加载次数等于需求未命中次数。传输时间估计值只计算需求加载字节：

\begin{equation}
    \label{equ:wrec-transfer-time}
    T_{\mathrm{transfer}}=
    \frac{N_{\mathrm{demand}}\cdot B_{\mathrm{expert}}}{b_{\mathrm{cg}}},
\end{equation}

其中，$N_{\mathrm{demand}}$ 是需求加载次数，$B_{\mathrm{expert}}$ 是单个专家对象大小，$b_{\mathrm{cg}}$ 是 CPU-GPU 有效带宽。该值表示不考虑计算与传输重叠时的专家权重传输时间估计，不等同于真实端到端延迟。对于包含预取的策略，模拟器额外统计预取传输量和预取浪费；正文主策略 WREC-H2 不启用预取，因此主结果中的 waste 为 $0$。

模拟器还计算 WREC 相对 LRU 的收益。设某一缓存预算下策略 $\pi$ 的传输时间估计值为 $T_\pi$，LRU 的结果为 $T_{\mathrm{LRU}}$，则策略相对 LRU 的收益为

\begin{equation}
    \label{equ:wrec-gain}
    G_{\pi}=\frac{T_{\mathrm{LRU}}-T_{\pi}}{T_{\mathrm{LRU}}}.
\end{equation}

同时，本文使用 $T_\pi-T_{\mathrm{Belady}}$ 描述在线策略与 Belady 离线上界之间的剩余差距。该指标用于判断 WREC 是否真正接近离线上界，而不仅仅是超过某一个在线基线。

\subsection{运行时接口与有限槽位原型}

除离线重放模拟器外，本文还实现了运行时 shadow 接口。该接口将在线 WREC 策略封装为轻量 sidecar 服务。推理框架在每个路由事件产生后向 sidecar 发送请求编号、层号、token 位置和 selected experts；sidecar 顺序维护 WREC 在线状态和 shadow resident set，并返回每个专家的 would-admit、would-bypass 和 would-evict 决策。该模式不控制真实专家加载，但可以验证 WREC 的状态更新和决策逻辑能否按运行时事件流在线执行。

在 vLLM 集成原型中，本文进一步实现了专家驻留管理接口。该接口支持三种层次。第一是 state-only 模式，只导出 routed expert 事件并消费 sidecar 决策，不改变真实专家权重放置。第二是 row-copy skeleton 模式，为特定专家保留 CPU backing，并在需要时把专家切片复制回 GPU 权重张量。第三是 finite-slot 模式，当 \texttt{WREC\_EXPERT\_RESIDENCY\_SLOT\_CAPACITY} 小于模型全局专家数时，将每层专家权重替换为有限 GPU slot 张量，并维护 logical expert 到 slot id 的 \texttt{expert\_map}。MoE kernel 通过该映射访问当前驻留的逻辑专家。

finite-slot 原型的作用是验证 WREC 决策能否进入真实专家驻留路径，而不是替代第 5 章的 trace-driven 主实验。该原型要求当前 batch 中每层实际访问的 unique experts 数量不超过 slot capacity，否则有限槽位无法保证正确执行。由于该路径同时受到 vLLM 调度、CPU offload、副本管理和 kernel 映射约束，第 5 章只将其作为运行时可行性验证，而不将其作为 WREC 算法主效果的唯一依据。

\subsection{本章小结}

本章介绍了 WREC 的方法设计和实现细节。WREC 首先从历史路由轨迹中构造窗口出现概率、窗口内期望路由次数和跨层转移概率，然后在在线事件流中维护 recent、request-local 和 token-layer 状态，并使用评分函数进行专家准入、绕过和换出决策。本文主策略采用 recent 权重为 $0$ 的 request/cross-layer WREC-H2，以适应 prefill total-budget 重放中的消融结果。最后，本章说明了离线重放模拟器和运行时 shadow/finite-slot 原型的实现边界。第 5 章将在这些实现基础上给出实验结果与分析。
