# 面向资源受限环境的 MoE 推理优化实验指南

## 1. 文档目的

这份文档用于指导毕业论文后续 10 天左右的实验实施，目标不是做一个“大而全”的推理系统，而是在现有硬件和时间约束下，完成一个可以运行、可以比较、可以写进论文的最小可行研究闭环。

本文档默认你的论文主题仍然是：

`面向资源受限环境的混合专家大模型多维资源协同推理优化方法`

但实验边界已经收缩为更可落地的版本：

- 只研究调度层优化，不重写底层执行系统
- 只固定一个主框架，不同时兼顾多个推理框架
- 只做一个主模型上的完整实验，不把时间分散到跨模型泛化
- 只保留两个核心贡献：
  - 长度感知与专家亲和联合调度
  - 轻量级风险约束准入

---

## 2. 当前建议的研究问题

建议你在后续论文正文中把研究问题统一表述为：

在单机资源受限的 MoE 在线推理场景中，面对请求长度不确定、KV Cache 持续增长和专家局部性变化等因素，如何仅通过调度层的轻量级改造，同时降低尾延迟并改善吞吐与资源稳定性？

这句话的好处是：

- 问题足够集中
- 不会把你逼到必须实现完整预取/驱逐系统
- 和你现有开题报告中的背景、意义、指标体系可以自然衔接

---

## 3. 实验总目标

你这次实验真正要回答的核心问题只有 3 个：

1. 在 MoE 推理场景中，单纯的 FCFS 是否明显落后于长度感知调度？
2. 在长度感知调度基础上，引入专家亲和信息后，是否能进一步改善尾延迟或吞吐？
3. 在联合调度基础上，再加入轻量级风险准入后，是否能降低高负载下的资源波动和极端延迟？

只要这 3 个问题能被实验回答清楚，你的论文主线就是完整的。

## 4. 建议采用的实验边界

### 4.1 主模型

优先使用工作区中已经存在的本地模型：

- `qwen1.5-MoE-A2.7B`

采用它而不是继续追求更大的 Mixtral，原因很现实：

- 本地已有模型文件，节省下载和环境时间
- 它是真正的 MoE 模型，足以支撑“专家局部性”研究
- 在 `2 x 3090Ti` 条件下更容易跑通可重复实验
- 对本科毕业论文来说，模型不是越大越好，实验闭环更重要

### 4.2 主框架

只选一个框架，建议优先顺序如下：

1. `vLLM`
2. 如果 `vLLM` 对当前模型支持不稳定，再考虑 Hugging Face 自建推理循环

不建议同时维护 `vLLM` 和 `LightLLM` 两套实现。

### 4.3 实验场景

只做单机场景，不做跨机，不做复杂分布式调度。

建议固定如下设置：

- 单机双 GPU
- 在线推理场景
- 连续批处理或近似连续批处理
- 模拟低、中、高三档负载

### 4.4 不做的内容

以下内容建议从论文实验部分中删除，或者降为“未来工作”：

- 跨模型泛化验证
- 专家转移概率矩阵的复杂建模
- 完整的专家预取-驻留-驱逐子系统
- 异步多线程调度模块
- 严格意义上的理论证明

## 5. 方法定义

### 5.0 当前实现边界

需要明确区分两种实验形态：

1. **客户端代理调度**
   - 当前 `scripts/runtime/log_vllm_requests.py` 中的 `length_only` 属于这一类。
   - 它是在请求发送到 vLLM OpenAI 兼容接口之前，先由客户端按照
     `input_tokens + target_max_new_tokens` 对请求顺序做重排。
   - 这种结果只能说明“外部到达顺序被改写后，服务端观测到的负载变化”，不能代表
     “调度策略已经嵌入 vLLM 内部调度器”的最终系统效果。

2. **服务端内嵌调度**
   - 这是论文期望的最终形态。
   - 调度逻辑应当以本地包、patch 或插件化模块的方式嵌入 vLLM 的服务端请求队列、
     batch 构造或 admission 路径中。
   - 最终论文主结果应优先基于这一形态采集。

因此，当前客户端版本的 `Length-only` 只适合作为 **proxy / preliminary**
实验，用来验证负载、日志链路和指标口径是否敏感；它不应直接作为最终方法结果。

### 5.1 基线方法

论文中至少保留 4 个策略：

1. `FCFS`
2. `Length-only`
3. `Affinity-only`
4. `Joint`

如果时间允许，再增加第 5 个：

5. `Joint + RiskAdmission`

### 5.2 联合调度策略

建议采用如下统一打分形式：

```text
Score(i) = α * LengthRank(i) + β * Affinity(i) - γ * Risk(i)
```

其中：

- `LengthRank(i)`：请求长度顺位得分
- `Affinity(i)`：请求与当前驻留专家集合的匹配程度
- `Risk(i)`：请求进入当前批次后可能带来的资源压力

建议先实现最简单可跑版本，再逐步细化参数。

### 5.3 LengthRank 的实现建议

不要一上来做复杂学习排序。

建议把 `LengthRank` 的实现明确拆成两个阶段：

### 第一阶段：近似长度排序

这是当前实验主线中的默认排序规则。实现上应分两步推进：先用客户端代理版本验证
指标是否有可观测差异，再迁移到 vLLM 服务端内嵌版本作为最终结果来源。

十天内最稳妥的方案是：

- 用输入长度作为先验
- 对 decode 阶段已运行请求，叠加当前已生成长度
- 用简单排序近似长度优先

可以采用如下近似形式：

```text
EstimatedLength(i) = InputTokens(i) + GeneratedTokensSoFar(i)
```

然后按 `EstimatedLength(i)` 从小到大排序。

也就是说，你论文里当前阶段可以写“长度感知排序”，而不必强行实现真正的 LTR 训练器。

这是完全可以接受的，但需要在文中诚实说明方法是“基于长度信息的近似顺位调度”。
如果结果来自客户端重排，还必须标注为代理实验，不能和服务端内嵌调度结果混用。

### 第二阶段：LTR 增强版本

当 `FCFS -> Length-only -> Affinity-only -> Joint -> Joint + RiskAdmission` 已经能够完整跑通，并且日志、画图、消融流程已经稳定后，再考虑把第一阶段中的近似长度排序替换为真正的 `LTR`。

此时你只需要替换 `LengthRank(i)` 的计算方式，而不需要重做整套实验框架。这样做有两个好处：

- 保证论文主线先成立
- 即使最终没有完成 LTR，已有实验结果仍然可以支撑论文正文

因此，本文默认将 `LTR` 视为“增强实验”而不是“主线前置条件”。

### 5.4 Affinity 的实现建议

这是整个方法里最关键、也最容易被做复杂的部分。建议你只做一个轻量可解释版本。

定义方式建议如下：

- 维护当前 GPU 驻留专家集合 `ResidentExperts`
- 对每个请求，记录最近若干步实际激活过的专家集合 `RecentExperts(i)`
- 计算两者的重合程度作为亲和分数

例如：

```text
Affinity(i) = |RecentExperts(i) ∩ ResidentExperts| / |RecentExperts(i) ∪ ResidentExperts|
```

如果新请求尚无历史专家信息，可以采取以下策略之一：

- 直接令 `Affinity(i)=0`
- 或在 prefill 后拿到第一轮路由信息再纳入 affinity 排序

推荐第一种，因为实现更简单。

### 5.5 Risk 的实现建议

不要做多步复杂预测，只做轻量风险估计。

建议风险项由以下部分构成：

- 当前 KV Cache 占用比例
- 当前活跃请求数
- 当前 resident experts 数量
- 当前批次加入该请求后可能额外增加的 token 负载

可以用线性组合：

```text
Risk(i) = w1 * KVPressure + w2 * ActiveLoad + w3 * ExpertSetSize + w4 * EstimatedTokenCost(i)
```

先做启发式规则，不要追求最优。

### 5.6 风险准入机制

准入机制建议和调度策略解耦。

简单说就是：

- 候选请求先打分排序
- 但不是分高就一定入批
- 如果预测资源压力接近阈值，则延迟接纳一部分请求

一个可直接实现的准入规则是：

```text
if CurrentKV + EstimatedKVGrowth + ExpertReserve > SafeLimit:
    delay admission
else:
    admit
```

这里的 `SafeLimit` 不要贴着显存上限取值，建议保留安全裕量。

## 6. 实验指标

### 6.1 核心指标

这几项是论文里必须要有的：

- 吞吐：`tokens/s`
- `TTFT`
- `TPOT`
- `P95 TTFT`
- `P95 TPOT`
- `P99 TPOT`
- GPU 显存峰值

### 6.2 建议补充指标

如果日志链路能做出来，建议再加：

- 活跃请求数时间序列
- 批次大小时间序列
- 驻留专家数量
- 专家切换次数
- resident set miss 次数

### 6.3 不要强求的指标

以下指标如果很难精确抓取，可以用代理变量替代，不要卡住进度：

- 精确 PCIe 流量
- 精确专家换入换出字节数
- 严格定义的重算次数

论文中可以写：

“由于当前实验平台与框架接口限制，本文采用专家切换次数和 resident miss 次数作为 I/O 压力代理指标。”

这是可以接受的。

## 7. 负载设计

### 7.1 数据来源

建议优先使用 ShareGPT 风格数据，或者自行构造长度分布接近真实请求的 prompt 集合。

如果来不及准备复杂语料，也可以采用“真实文本 + 合成长度分布”的方案：

- 收集一批真实 prompt
- 控制输入长度分布
- 控制最大生成长度

### 7.2 三档负载

建议固定三档：

- 低负载：系统基本无排队
- 中负载：系统有轻度排队和资源竞争
- 高负载：系统接近瓶颈，便于观察尾延迟和稳定性

不要做太多负载档位，三档足够。

### 7.3 控制变量

你后续实验必须尽量保证以下内容固定：

- 模型固定
- 硬件固定
- 框架版本固定
- prompt 数据固定
- 生成长度上限固定
- 随机种子尽量固定

否则实验波动会很难解释。

## 8. 实现顺序

### 阶段一：跑通主链路

目标：

- 跑通本地 MoE 模型推理
- 打通日志采集链路
- 能输出基础时延和吞吐

完成标准：

- 至少能稳定完成一轮基准推理
- 能保存每个请求的开始时间、首 token 时间、结束时间
- 能记录显存使用

### 阶段二：完成 FCFS 与客户端代理 Length-only

目标：

- 实现 `FCFS`
- 实现基于近似长度排序的客户端代理 `Length-only`

完成标准：

- 两个基线都能重复运行
- 同一负载下能看出差异
- 初步图表能画出来
- 所有客户端代理结果都明确标注为 preliminary / proxy

### 阶段三：迁移到 vLLM 服务端内嵌调度

目标：

- 找到 vLLM 服务端请求队列、batch 构造或 admission 相关的最小改造点
- 将 `Length-only` 从客户端重排迁移为服务端内部调度逻辑
- 保持原有日志与分析脚本尽量不变

完成标准：

- FCFS 与服务端内嵌 `Length-only` 能在同一套负载下对比
- 运行记录能说明调度逻辑是在 vLLM 服务端生效，而不是客户端预排序
- 这组结果可以作为论文主结果的第一个方法对照

### 阶段四：完成专家亲和调度

目标：

- 能记录请求的最近专家集合
- 能维护 resident experts
- 能实现 `Affinity-only` 和 `Joint`

完成标准：

- 至少在中高负载下观察到联合策略与长度策略的差异
- 无论收益大或小，都要能解释

### 阶段五：完成风险准入

目标：

- 实现简单版本 `Joint + RiskAdmission`

完成标准：

- 在高负载下至少看到一种效果：
  - 显存峰值下降
  - P99 TPOT 改善
  - 系统波动减小

### 阶段六：跑正式实验

目标：

- 固定全部参数
- 统一跑所有策略
- 输出可复现日志
- 使用服务端内嵌调度实现，而不是客户端代理重排

完成标准：

- 每个策略每个负载至少跑 3 次
- 取均值和方差或标准差

### 阶段七：可选的 LTR 增强

目标：

- 在主实验已经闭环的前提下，尝试将近似长度排序替换为真实 `LTR`

完成标准：

- 不影响原有实验链路
- 能与第一阶段的 `Length-only` 直接做对比
- 若实现失败，不影响论文主体结论

## 12. 可能遇到的主要风险

### 风险 1：框架对本地 MoE 模型支持不稳定

应对：

- 优先验证最小推理链路
- 如果框架直接不支持，不要死磕，及时降级到更简单实现

### 风险 2：拿不到精确专家迁移数据

应对：

- 改用代理指标
- 在论文中明示限制

### 风险 3：联合策略收益不明显

应对：

- 重点分析高负载场景
- 重点看尾延迟而不是只看平均值
- 强调“稳定性收益”而不只是吞吐收益

### 风险 4：风险准入拖慢整体吞吐

应对：

- 这是正常现象
- 论文里应当写成“以小幅吞吐代价换取尾延迟和资源稳定性改善”

### 风险 5：十天内实现不完

应对：

- 保底顺序必须是：
  - `FCFS`
  - `Length-only`
  - `Joint`
- `RiskAdmission` 可以作为加分项，但不是唯一生死线
